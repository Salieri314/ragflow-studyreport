## 第五部分：DSL 被 Canvas 加载之后，如何变成可执行对象。

也就是这一步：

```ABAP
canvas = Canvas(dsl_str, tenant_id, canvas_id=agent_id)
```

这行代码看起来很短，但它背后做了很多事情。

核心代码在：

agent/canvas\.py \(line 287\)

```Python
class Canvas(Graph):

    def __init__(self, dsl: str, tenant_id=None, task_id=None, canvas_id=None, custom_header=None):
        self.globals = {...}
        self.variables = {}
        super().__init__(dsl, tenant_id, task_id, custom_header=custom_header)
        self._id = canvas_id
```

这里可以分成三步理解。

---

### 第一步：初始化系统变量

Canvas 一开始会准备一组全局变量：

```Python
self.globals = {
    "sys.query": "",
    "sys.user_id": tenant_id,
    "sys.conversation_turns": 0,
    "sys.files": [],
    "sys.history": [],
    "sys.date": ...
}
```

这些变量是运行时公共上下文。

比如：

```ABAP
sys.query
  当前用户的问题

sys.user_id
  当前用户 / 租户 id

sys.files
  用户上传的文件

sys.history
  对话历史

sys.conversation_turns
  当前是第几轮对话
```

所以 Canvas 创建时，先准备一个“运行环境”。

这就像程序启动时先准备环境变量。

---

### 第二步：调用 Graph 解析 DSL

Canvas 继承自 Graph，所以它会调用：

```ABAP
super().**init**(dsl, tenant_id, task_id, custom_header=custom_header)
```

父类 Graph\.**init**\(\) 在：

agent/canvas\.py \(line 84\)

核心代码：

```ABAP
self.dsl = normalize_chunker_dsl(json.loads(dsl))
self.load()
```

这里做两件事：

```ABAP
1. json.loads(dsl)
   把 DSL 字符串转成 Python dict

2. normalize_chunker_dsl(...)
   做 DSL 兼容和格式迁移

3. self.load()
   真正加载组件
```

所以：

```ABAP
Canvas(dsl_str)
   ↓
Graph.__init__()
   ↓
解析 JSON
   ↓
规范化 DSL
   ↓
load components
```

其中 normalize\_chunker\_dsl 可以理解成兼容旧版本 DSL 的迁移函数。

因为产品升级后，DSL 结构可能变过。为了老的画布还能跑，就需要在加载时做一次规范化。

---

### **第三步：加载 components**

最关键的是 Graph\.load\(\)。

代码在：

agent/canvas\.py \(line 96\)

```Python
self.components = self.dsl["components"]

for k, cpn in self.components.items():
    param = component_class(cpn["obj"]["component_name"] + "Param")()
    cpn["obj"]["params"]["custom_header"] = self.custom_header
    param.update(cpn["obj"]["params"])
    param.check()
    cpn["obj"] = component_class(cpn["obj"]["component_name"])(self, k, param)

self.path = self.dsl["path"]
```

这段是第五部分最重要的代码。

我们拆开看。

---

假设 DSL 里有一个组件：

```JSON
{
  "retrieval_0": {
    "obj": {
      "component_name": "Retrieval",
      "params": {
        "kb_ids": ["xxx"],
        "top_k": 8
      }
    },
    "downstream": ["generate_0"],
    "upstream": ["begin"]
  }
}
```

加载时，第一步取出：

```ABAP
component_name = "Retrieval"
```

然后拼出参数类名：

```ABAP
"Retrieval" + "Param"
```

也就是：

```ABAP
RetrievalParam
```

然后用：

```ABAP
component_class("RetrievalParam")()
```

创建参数对象。

---

接着执行：

```ABAP
param.update(cpn["obj"]["params"])
```

这一步是把 DSL 里的 JSON 参数灌进参数对象。

也就是从：

```ABAP
{
  "kb_ids": ["xxx"],
  "top_k": 8
}
```

变成：

```ABAP
RetrievalParam.kb_ids = ["xxx"]
RetrievalParam.top_k = 8
```

参数更新的基础逻辑在：

agent/component/base\.py \(line 127\)

```ABAP
def update(self, conf, allow_redundant=False):
```

它会递归地把 dict 里的字段写到 Param 对象里。

---

然后执行：

```ABAP
param.check()
```

这一步是参数校验。

每个组件自己的 Param 类可以定义自己的 check\(\)。

比如 Begin 节点在：

agent/component/begin\.py \(line 20\)

```ABAP
class BeginParam(UserFillUpParam):
```

它的校验是：

```ABAP
agent/component/begin.py (line 30)
```

```Python
def check(self):
    self.check_valid_value(
        self.mode,
        "The 'mode' should be either `conversational` or `task`",
        ["conversational", "task", "Webhook"]
    )
```

也就是说，Begin 节点的运行模式只能是：

```ABAP
conversational
task
Webhook
```

如果 DSL 里写了一个非法值，比如：

```ABAP
mode = "abc"
```

那 Canvas 初始化阶段就会报错。

这说明 RAGFlow 不是等执行到节点才发现配置错了，而是在加载 DSL 时就先检查一遍。

---

### 最后一步是创建真正的组件对象：

```ABAP
cpn["obj"] = component_class(cpn["obj"]["component_name"])(self, k, param)
```

继续用 Retrieval 举例：

```ABAP
component_name = Retrieval
```

那么它会创建：

```ABAP
Retrieval(self, "retrieval_0", param)
```

这里的 self 就是当前 Canvas。

这意味着每个组件对象都能反过来访问 Canvas，例如：

```ABAP
读取全局变量
读取上游组件输出
写入自己的输出
获取租户 id
记录引用信息
```

所以组件不是孤立运行的，它们都挂在同一个 Canvas 上。

---

这一段代码的核心变化是：

```ABAP
加载前：
components[id]["obj"] 是 JSON

加载后：
components[id]["obj"] 是 Python 组件对象
```

也就是：

```ABAP
DSL 静态定义
   ↓
Canvas.load()
   ↓
可执行组件实例
```

---

Canvas\.load\(\) 之后，Canvas 自己还会加载一些运行状态。

代码在：

agent/canvas\.py \(line 300\)

```Python
def load(self):
    super().load()
    self.history = self.dsl["history"]
    ...
    self.retrieval = self.dsl["retrieval"]
    self.memory = self.dsl.get("memory", [])
```

这说明 Canvas 不只加载节点，还加载：

```ABAP
history
  历史对话

globals
  全局变量

variables
  用户自定义变量

retrieval
  检索引用

memory
  记忆
```

所以完整加载过程是：

```ABAP
Graph.load()
  加载组件结构

Canvas.load()
  加载对话状态、变量、引用、记忆
```

---

### 第五部分可以总结成这样：

```ABAP
Canvas(dsl)
   ↓
初始化 sys.query / sys.user_id / sys.files 等系统变量
   ↓
Graph.__init__()
   ↓
json.loads(dsl)
   ↓
normalize_chunker_dsl()
   ↓
Graph.load()
   ↓
遍历 dsl["components"]
   ↓
根据 component_name 找到 Param 类
   ↓
param.update(params)
   ↓
param.check()
   ↓
根据 component_name 创建真正组件对象
   ↓
Canvas.load()
   ↓
恢复 history / globals / variables / retrieval / memory
```

一句话说：

> 第五部分就是 RAGFlow 把 DSL 从“静态 JSON 配置”加载成“可执行 Canvas 对象”的过程。它会先解析 DSL，再把每个 component 转成对应的 Python 组件实例，同时完成参数灌入、参数校验、变量恢复和运行状态初始化。
