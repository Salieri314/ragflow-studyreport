## 第四部分：运行入口

**用户点击运行 / 对话 / 调试之后，后端怎么进入 Canvas 执行。**

这一步开始，DSL 就不再只是“保存的数据”了，而是要被加载成真正的执行对象。

核心入口有几个，但最重要的是这个：

api/apps/restful\_apis/agent\_api\.py \(line 1129\)

```Python
@manager.route("/agents/chat/completions", methods=["POST"])
async def agent_chat_completion(...)
```

也就是说，Agent Canvas 的运行、调试、对话，大部分都会走：

```ABAP
POST /agents/chat/completions
```

前端发送消息的地方在：

web/src/pages/agent/chat/use\-send\-agent\-message\.ts \(line 245\)

它会用 SSE 请求这个接口：

```ABAP
useSendMessageBySSE(url || api.agentChatCompletion)
```

所以普通 Agent 对话链路是：

```ABAP
用户在 Agent Chat 里输入问题
   ↓
前端发送 SSE 请求
   ↓
POST /agents/chat/completions
   ↓
agent_chat_completion()
   ↓
加载 DSL
   ↓
Canvas.run()
   ↓
SSE 返回事件流
```

---

这里先看后端入口做的第一件事。

在 agent\_chat\_completion\(\) 里，它会从请求体里拿：

```ABAP
req = await get_request_json()
agent_id = agent_id or req.get("agent_id")
session_id = req.get("session_id")
openai_compatible = bool(req.get("openai-compatible", False))
```

这几个字段非常关键：

```ABAP
agent_id:
  要运行哪个画布

session_id:
  是不是继续一个已有会话

openai-compatible:
  是否用 OpenAI 风格接口返回
```

所以运行入口首先要判断：

```ABAP
这是一次新的运行？
还是继续已有 session？
还是 OpenAI-compatible 调用？
```

---

### 第一种情况：普通新运行，没有 session\_id。

这通常发生在你点调试、运行，或者发起一次新的 Agent 对话。

后端会先检查权限：

```ABAP
UserCanvasService.accessible(agent_id, tenant_id)
```

然后加载运行副本：

```ABAP
replica_payload = CanvasReplicaService.load_for_run(...)
```

代码位置在：

api/apps/restful\_apis/agent\_api\.py \(line 1272\)

如果 Redis 里没有运行副本，它会用数据库里的 user\_canvas\.dsl 初始化一份：

```ABAP
CanvasReplicaService.bootstrap(...)
```

所以这里的逻辑是：

```ABAP
优先用 Redis canvas replica
如果没有，就从 user_canvas.dsl 创建 replica
```

这和上一部分保存逻辑接上了：

```ABAP
保存时:
  user_canvas.dsl -> Redis replica

运行时:
  Redis replica -> Canvas
```

---

拿到 DSL 之后，会创建 Canvas 对象：

```ABAP
canvas = Canvas(dsl_str, str(tenant_id), canvas_id=agent_id, custom_header=custom_header)
```

这就是从“DSL 文本”进入“执行引擎”的关键一步。

你可以把它理解为：

```ABAP
DSL 是流程定义文件
Canvas 是流程执行器
```

类似：

```ABAP
配置文件 -> 程序对象
```

还没到真正执行节点，但已经开始把 DSL 变成 Python 里的运行时对象。

---

### 第二种情况：继续已有 session\_id。

如果请求里带了 session\_id，后端不会简单重新读取最新 DSL。

它会先查：

```ABAP
API4ConversationService.get_by_id(session_id)
```

然后检查：

```ABAP
conv.dialog_id == agent_id
```

确认这个 session 确实属于当前 agent。

如果是普通 Agent 会话，真正继续执行的代码在：

api/db/services/canvas\_service\.py \(line 313\)

```ABAP
async def completion(tenant_id, agent_id, session_id=None, **kwargs):
```

如果有 session\_id：

```ABAP
canvas = Canvas(conv.dsl, tenant_id, agent_id, canvas_id=agent_id, custom_header=custom_header)
```

注意这里用的是：

```ABAP
conv.dsl
```

也就是会话里保存的那份 DSL。

这点很重要。

因为一个用户开始会话后，你后来可能修改了画布。如果继续这个旧 session，RAGFlow 更倾向于用这个 session 当时保存的 DSL，而不是直接切到最新画布。

所以：

```ABAP
新会话:
  用当前 DSL / 发布 DSL / replica

旧会话:
  用 API4Conversation.dsl
```

这样可以避免一个会话聊到一半，画布突然变了，导致上下文和流程不一致。

---

### 第三种情况：发布模式 release。

有些入口可以带：

```ABAP
release=true
```

这时不是用 user\_canvas\.dsl，而是用发布版本。

代码在：

api/db/services/canvas\_service\.py \(line 294\)

```ABAP
get_agent_dsl_with_release(agent_id, release_mode=True)
```

如果 release\_mode=True：

```ABAP
released_version = UserCanvasVersionService.get_latest_released(agent_id)
dsl = released_version.dsl
```

也就是：

```ABAP
release=false 或默认:
  用当前编辑版 DSL

release=true:
  用最新发布版 DSL
```

这就是 RAGFlow 区分“编辑调试”和“对外发布”的方式。

---

### 第四种情况：Dataflow 运行入口。

RAGFlow 的画布不只有 Agent Canvas，还有 Dataflow Canvas。

前端 Dataflow 运行在：

```ABAP
web/src/pages/agent/hooks/use-run-dataflow.ts (line 16)
```

它也调用：

```ABAP
api.agentChatCompletion
```

也就是说，Dataflow 和 Agent 入口共用 /agents/chat/completions。

但后端会判断：

```ABAP
if cvs.canvas_category == CanvasCategory.DataFlow:
```

如果是 Dataflow，就不创建 agent\.canvas\.Canvas，而是创建：

```ABAP
from rag.flow.pipeline import Pipeline
```

然后把任务丢到 Dataflow 队列里：

```ABAP
queue_dataflow(...)
```

所以这里有分叉：

```ABAP
Agent Canvas:
  Canvas(dsl).run(...)

Dataflow Canvas:
  Pipeline(dsl)
  queue_dataflow(...)
```

这解释了为什么同一个接口可以运行两类画布。

---

### 第五种情况：OpenAI\-compatible 调用。

agent\_chat\_completion\(\) 里还有一段：

```ABAP
if openai_compatible:
    ...
    completion_openai(...)
```

这表示 RAGFlow 可以把 Agent Canvas 包装成类似 OpenAI Chat Completions 的接口形式。

普通模式返回的是 RAGFlow 自己的事件：

```ABAP
message
workflow_started
node_started
node_finished
...
```

OpenAI\-compatible 模式会转换成 OpenAI 风格响应。

所以这个入口同时支持：

```ABAP
RAGFlow 原生事件流
OpenAI-compatible API
```

---

### 最后看真正执行的地方。

在：

api/db/services/canvas\_service\.py \(line 313\)

核心代码是：

```ABAP
async for ans in canvas.run(**run_kwargs):
    yield "data:" + json.dumps(ans, ensure_ascii=False) + "\n\n"
```

这说明 Canvas 执行结果是一个异步事件流。

也就是说，后端不是等整个流程全部跑完再一次性返回，而是：

```ABAP
Canvas 每执行出一个事件
   ↓
yield 一个 SSE data
   ↓
前端实时显示
```

这就是为什么你在界面上能看到节点逐步运行、日志逐步出现。

---

这一部分可以总结成一张图：

```ABAP
前端点击运行 / 发送消息
   ↓
POST /agents/chat/completions
   ↓
agent_chat_completion()
   ↓
判断模式
   ├─ 新运行：加载 Redis replica / user_canvas.dsl
   ├─ 旧 session：加载 API4Conversation.dsl
   ├─ release=true：加载 user_canvas_version 最新发布 DSL
   ├─ Dataflow：进入 Pipeline + queue_dataflow
   └─ OpenAI-compatible：转换成 OpenAI 风格输出
   ↓
Agent Canvas:
   Canvas(dsl)
   ↓
canvas.run(...)
   ↓
SSE 事件流返回前端
```

一句话总结：

> 第四部分是 RAGFlow 把“保存好的 DSL”真正送入执行入口的过程。普通 Agent 运行会创建 Canvas，Dataflow 会创建 Pipeline，新运行通常从 Redis replica 或当前 DSL 加载，旧 session 则从会话保存的 DSL 恢复，发布模式会使用 user\_canvas\_version 里的 release 版本。
> 
> 





### Canvas是什么？？？

相信肯定有同学对于Canvas是什么没有基础的了解。

在 RAGFlow 里，Canvas 不是前端那个“画布 UI”。

它是后端 Python 里的一个流程执行器。

代码在：agent/canvas\.py \(line 285\)

```ABAP
class Canvas(Graph):
```

也就是说：

```ABAP
前端 Canvas:
  用户拖节点、连线的可视化界面

后端 Canvas:
  读取 DSL，然后真正执行这些节点的运行时对象
```

你可以把它理解成：

```ABAP
前端画布 = 画流程图的编辑器
DSL = 流程图保存出来的配置文件
后端 Canvas = 按照 DSL 跑流程的解释器/执行器
```

---

它的输入是一份 DSL。

在运行入口里会看到类似代码：

```ABAP
canvas = Canvas(dsl_str, tenant_id, agent_id, canvas_id=agent_id)
```

也就是说，Canvas 初始化时拿到的是：

```ABAP
dsl_str:
  前端保存下来的流程定义

tenant_id:
  当前租户 / 用户

canvas_id:
  当前画布 id
```

然后 Canvas 会把 DSL 变成一组真正可执行的组件对象。



它的继承关系是：

```ABAP
Graph
  ↓
Canvas
```



Graph 在：

agent/canvas\.py \(line 43\)

```ABAP
class Graph:
```

Graph 负责比较底层的事情：

```ABAP
解析 DSL
加载 components
根据 component_name 找组件类
初始化组件参数
维护 path
提供变量读取方法
```

Canvas 在 Graph 的基础上，增加了 Agent 运行需要的东西：

```ABAP
globals
variables
history
retrieval
memory
run()
```

所以可以这么分：

```ABAP
Graph:
  管流程图结构

Canvas:
  管一次具体运行
```

---

#### Canvas 初始化时，先设置一组全局变量。

代码在：

```Python
self.globals = {
    "sys.query": "",
    "sys.user_id": tenant_id,
    "sys.conversation_turns": 0,
    "sys.files": [],
    "sys.history": [],
    "sys.date": ...
}
```

这些就是系统级变量。

比如用户问了：帮我查一下员工报销制度

运行时会放进：

```ABAP
sys.query = "帮我查一下员工报销制度"
```

如果用户上传了文件，会放进：

```ABAP
sys.files = [...]
```

如果是多轮对话，会更新：

```ABAP
sys.history
sys.conversation_turns
```

所以 Canvas 不只是执行节点，它还维护这次流程运行的上下文。

---

然后 Canvas 会调用父类 Graph\.**init**\(\)。

父类里有一个很关键的步骤：

agent/canvas\.py \(line 84\)

```ABAP
self.dsl = normalize_chunker_dsl(json.loads(dsl))
self.load()
```

这里做两件事：

```ABAP
1. 把 DSL 字符串解析成 Python dict
2. 调用 load() 加载组件
```

load\(\) 在：

agent/canvas\.py \(line 96\)

核心代码是：

```Python
self.components = self.dsl["components"]

for k, cpn in self.components.items():
    param = component_class(cpn["obj"]["component_name"] + "Param")()
    param.update(cpn["obj"]["params"])
    param.check()
    cpn["obj"] = component_class(cpn["obj"]["component_name"])(self, k, param)
```

这段代码非常重要。

它说明 DSL 里的每一个组件，都会被转换成真正的 Python 对象。

比如 DSL 里有：

```ABAP
component_name = Retrieval
```

后端会动态找到：

```ABAP
RetrievalParam
Retrieval
```

然后创建：

```ABAP
Retrieval(canvas, component_id, param)
```

所以 Canvas 的第一个核心职责就是：

> 把 DSL 里的静态组件定义，变成 Python 里的可执行组件对象。
> 
> 

---

简单举个例子。

前端 DSL 里可能有：

```YAML
components:
  retrieval_0:
    obj:
      component_name: Retrieval
      params:
        kb_ids: [...]
        top_k: 8
    downstream:
      - generate_0
```

Canvas\.load\(\) 后，会变成：

```ABAP
components:
  retrieval_0:
    obj: Retrieval 对象
    downstream:
      - generate_0
```

也就是说：

```ABAP
保存时:
  obj 是 JSON

运行时:
  obj 是 Python 组件实例
```

这是 Canvas 最关键的变化。

---

#### Canvas 的第二个职责是：维护执行路径 path。

DSL 里面有一个字段：

```ABAP
path
```

运行时 Canvas 会根据当前执行情况不断更新它。

比如一开始：

```ABAP
path = ["begin"]
```

执行 Begin 之后，根据下游节点继续追加：

```ABAP
path = ["begin", "retrieval_0"]
```

再往后：

```ABAP
path = ["begin", "retrieval_0", "generate_0", "message_0"]
```

如果遇到分类节点、Switch 节点，路径可能不是固定的，而是根据输出结果决定走哪个分支。

所以 Canvas 不是一次性把所有节点都跑掉，而是维护一条动态执行路径。

---

#### Canvas 的第三个职责是：**运行流程并产生事件流**。

代码在：

agent/canvas\.py \(line 375\)

运行时它会不断 yield 事件，比如：

```ABAP
workflow_started
node_started
node_finished
message
message_end
user_inputs
workflow_finished
```

这就是前端为什么能实时看到节点执行状态。

本质上是：

```ABAP
Canvas.run()
   ↓
每跑到一个阶段 yield 一个事件
   ↓
后端 SSE 返回给前端
   ↓
前端实时渲染运行过程
```

---

#### Canvas 的第四个职责是：变量解析。

在画布里，你可能会配置类似：

```ABAP
{{sys.query}}
{{retrieval_0@chunks}}
{{generate_0@content}}
{{env.api_key}}
```

这些变量不是前端替换的，而是运行时 Canvas 负责解析。

相关函数在：

agent/canvas\.py \(line 165\)

```ABAP
get_variable_value(...)
```

它支持三类变量：

```ABAP
sys.xxx
  系统变量，比如 sys.query

env.xxx
  用户配置的全局变量

component_id@field
  某个组件的输出，比如 retrieval_0@chunks
```

所以 Canvas 也是整个流程的数据访问中心。

---

#### Canvas 的第五个职责是：保存运行状态。

它维护：

```ABAP
history:
  对话历史

retrieval:
  检索引用

memory:
  记忆

error:
  错误信息

task_id:
  当前任务 id
```

运行结束后，外层会把这些写回会话。

比如在：

api/db/services/canvas\_service\.py \(line 313\)

运行后会做：

```ABAP
conv.reference = canvas.get_reference()
conv.errors = canvas.error
conv.dsl = str(canvas)
```

这里的 str\(canvas\) 会把运行后的 Canvas 状态重新序列化成 DSL。

所以它不只是执行，还会把执行后的状态保存回会话。

---

#### 总结

```ABAP
前端画布
  nodes + edges + form
        ↓
DSL
  graph + components + globals
        ↓
Canvas(dsl)
        ↓
Canvas.load()
  把 components 转成 Python 组件对象
        ↓
Canvas.run()
  按 path 执行节点
        ↓
输出事件流
  workflow_started
  node_started
  node_finished
  message
  workflow_finished
        ↓
保存 history / retrieval / error / dsl
```

一句话说：

> Canvas 是 RAGFlow Agent 画布的后端运行时解释器。它读取 DSL，把 DSL 里的组件变成 Python 对象，维护全局变量、执行路径、历史记录和引用信息，然后按节点顺序运行整个流程，并通过事件流把执行过程返回给前端。
> 
>
