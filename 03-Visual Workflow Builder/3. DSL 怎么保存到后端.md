## 第三部分：DSL 怎么保存到后端，以及草稿版 / 发布版 / 运行副本之间是什么关系

上一部分我们讲到，前端会生成：

```ABAP
dsl:
  graph
  components
  globals
  variables
  ...
```

然后调用：

web/src/pages/agent/hooks/use\-save\-graph\.ts \(line 17\)

```ABAP
saveGraph()
  -> buildDslData()
  -> setAgent(params)
```

这里的 params 大概是：

```ABAP
{
  id,
  title,
  dsl: buildDslData(...),
  release
}
```

也就是说，前端保存 Agent Canvas 时，真正提交给后端的是完整 DSL。

---

前端请求会进入：

web/src/hooks/use\-agent\-request\.ts \(line 390\)

这里的逻辑是：

```ABAP
如果有 agentId:
  updateAgent(agentId, ...)
否则:
  createAgent(...)
```

也就是说：

```ABAP
新建画布 -> POST /agents
更新画布 -> PUT /agents/<agent_id>
```

对应后端在：

api/apps/restful\_apis/agent\_api\.py \(line 614\)

新建接口：

```Python
@manager.route("/agents", methods=["POST"])
async def create_agent(...)
```

更新接口在：

api/apps/restful\_apis/agent\_api\.py \(line 867\)

```Python
@manager.route("/agents/<agent_id>", methods=["PUT"])
async def update_agent(...)
```

---

这里最重要的不是接口本身，而是它保存了不止一份 DSL。

### RAGFlow 这里至少有三层保存：

```ABAP
1. user_canvas.dsl
   当前画布主记录

2. user_canvas_version.dsl
   版本快照

3. Redis canvas replica
   运行副本
```

先看数据库模型。

在：

api/db/db\_models\.py \(line 1057\)

有 UserCanvas：

```ABAP
class UserCanvas(DataBaseModel):
    id
    user_id
    title
    release
    canvas_category
    dsl
```

它对应表：

```ABAP
user_canvas
```

这张表可以理解为：

```ABAP
当前这个 Agent / Dataflow 画布的主表。
```

也就是说，当你在页面里打开某个 Agent，大部分情况下看到的是 user\_canvas\.dsl。

---

### 第二张表是：

api/db/db\_models\.py \(line 1089\)

```ABAP
class UserCanvasVersion(DataBaseModel):
    user_canvas_id
    title
    release
    dsl
```

对应表：

```ABAP
user_canvas_version
```

这张表可以理解为：

> 某个画布的历史版本 / 发布版本快照。
> 
> 

所以它和 user\_canvas 的关系是：

```ABAP
user_canvas
  一个画布当前主记录

user_canvas_version
  这个画布的多个版本快照
```

类似：

```ABAP
user_canvas.id = A001

user_canvas_version:
  version 1 -> A001 的旧 DSL
  version 2 -> A001 的新 DSL
  version 3 -> A001 的发布版 DSL
```

---

### **创建 Agent 的时候，后端会做三件事。**

代码在：

api/apps/restful\_apis/agent\_api\.py \(line 614\)

第一，标准化 DSL：

```ABAP
req["dsl"] = CanvasReplicaService.normalize_dsl(req["dsl"])
```

这一步不是执行业务，而是确保 DSL 是合法 JSON，并且做一些兼容性规范化。

第二，保存到 user\_canvas：

```ABAP
UserCanvasService.save(**req)
```

第三，保存版本快照：

```ABAP
UserCanvasVersionService.save_or_replace_latest(
    user_canvas_id=req["id"],
    dsl=req["dsl"],
    release=req.get("release"),
)
```

第四，同步 Redis 运行副本：

```ABAP
CanvasReplicaService.replace_for_set(...)
```

所以创建链路是：

```ABAP
POST /agents
   ↓
normalize_dsl
   ↓
写 user_canvas.dsl
   ↓
写 user_canvas_version.dsl
   ↓
写 Redis canvas replica
```

---

更新 Agent 时也是类似的。

代码在：

api/apps/restful\_apis/agent\_api\.py \(line 867\)

核心逻辑：

```ABAP
UserCanvasService.update_by_id(agent_id, req)
```

这会更新 user\_canvas。

如果请求里有 dsl，还会继续：

```ABAP
UserCanvasVersionService.save_or_replace_latest(...)
CanvasReplicaService.replace_for_set(...)
```

所以更新链路是：

```ABAP
PUT /agents/<agent_id>
   ↓
normalize_dsl
   ↓
更新 user_canvas.dsl
   ↓
更新/新增 user_canvas_version.dsl
   ↓
刷新 Redis 运行副本
```

---

这里有一个关键问题：每次保存都会新建一个版本吗？

答案是：不是。

版本逻辑在：api/db/services/user\_canvas\_version\.py \(line 129\)

函数：

```ABAP
save_or_replace_latest(...)
```

它的规则大概是：

```ABAP
如果最新版本的 DSL 和当前 DSL 一样：
  更新最新版本

如果 DSL 内容真的变了：
  新增一个版本快照

如果最新版本是 release=True，而这次不是发布：
  新建一个草稿版本，保护已发布版本
```

这点很重要。

因为 RAGFlow 不希望你继续编辑草稿时，把已经发布出去的版本覆盖掉。

所以它有一个保护逻辑：

```ABAP
发布版 DSL
  不轻易被普通保存覆盖

普通保存
  可能成为新的草稿版本
```

---

release 是什么意思？

你可以理解成：

```ABAP
release = False
  草稿版 / 编辑版

release = True
  发布版 / 对外可用版
```

在运行时，如果请求指定要用发布版，后端会从 user\_canvas\_version 找最新发布版本。

代码在：

api/db/services/canvas\_service\.py \(line 294\)

```ABAP
def get_agent_dsl_with_release(agent_id, release_mode=False, tenant_id=None):
    if release_mode:
        released_version = UserCanvasVersionService.get_latest_released(agent_id)
        dsl = released_version.dsl
    else:
        dsl = cvs.dsl
```

翻译一下：

```ABAP
普通运行:
  用 user_canvas.dsl

发布模式运行:
  用 user_canvas_version 里最新 release=True 的 DSL
```

所以它不是永远只用主表。

它分两种情况：

```ABAP
编辑 / 调试 / 普通运行
  当前 DSL

发布访问 / release 模式
  最新发布 DSL
```

---

### 第三份：Redis 运行副本。

代码在：

api/apps/services/canvas\_replica\_service\.py \(line 20\)

它的作用是：

```ABAP
给运行时保存一份临时 DSL 副本
```

更新 Agent 时会调用：

```ABAP
CanvasReplicaService.replace_for_set(...)
```

这个函数会把 DSL 写入 Redis，key 类似：

```ABAP
canvas:replica:<canvas_id>:<tenant_id>:<runtime_user_id>
```

它还有 TTL：

```ABAP
TTL_SECS = 3 * 60 * 60
```

也就是 3 小时。

这说明 Redis 里的这份不是永久主数据，而是运行期副本。

你可以这样理解：

```ABAP
MySQL:
  保存长期数据

Redis:
  保存运行时临时副本 / 当前执行状态

```

---

所以第三部分的完整保存结构是：

```ABAP
前端 buildDslData()
   ↓
setAgent()
   ↓
POST /agents 或 PUT /agents/<id>
   ↓
normalize_dsl
   ↓
user_canvas.dsl
   当前画布主记录
   ↓
user_canvas_version.dsl
   版本快照 / 发布版保护
   ↓
Redis canvas replica
   运行时副本
```

用一句话总结：

> RAGFlow 的 DSL 不是只保存一份。user\_canvas\.dsl 保存当前画布，user\_canvas\_version\.dsl 保存版本和发布快照，Redis replica 保存运行时副本。这样它既能支持画布编辑，又能支持发布版本隔离，还能让运行时有一份临时可更新的状态。
> 
>
