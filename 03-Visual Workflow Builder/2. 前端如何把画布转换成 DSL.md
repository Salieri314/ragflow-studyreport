## 第二部分：前端如何把画布转换成 DSL

上一部分我们说，画布里只有两类核心数据：

```ABAP
nodes = 节点
edges = 连线
```

但后端执行时，不想直接对着 ReactFlow 的画布数据跑。因为 ReactFlow 的 nodes/edges 更偏“前端绘图结构”，里面有位置、样式、拖拽状态、handle 信息。

所以 RAGFlow 会在保存前，把它转换成一份更适合后端执行的 DSL。

核心入口是：web/src/pages/agent/hooks/use\-build\-dsl\.ts \(line 15\)

关键函数：

```ABAP
const buildDslData = useCallback(...)
```

它做的事情可以概括成：

```ABAP
前端画布状态
nodes + edges
   ↓
过滤掉占位节点
   ↓
生成 graph
   ↓
生成 components
   ↓
合成完整 DSL
```

---

DSL 里面最重要的是两个部分：

```ABAP
return {
  ...data.dsl,
  ...globalVariables,
  graph: { nodes: filteredNodes, edges: filteredEdges },
  components: dslComponents,
};
```

也就是说，一个 DSL 里同时保存两套结构：

```ABAP
graph:
  给前端恢复画布用

components:
  给后端执行流程用
```

这个区别非常重要。

你可以这样理解：

```ABAP
graph 是“画出来的图”
components 是“跑起来的流程”
```

---

### 先看 graph。

代码里：

```ABAP
graph: { nodes: filteredNodes, edges: filteredEdges }
```

它基本保留了 ReactFlow 的画布信息。

所以 graph 主要负责：

```ABAP
下次打开页面时，节点还在原来的位置
连线还保持原样
节点样式、名称、form 还能恢复
```

也就是说，graph 更偏前端。

如果没有 graph，后端也许还能执行，但前端不知道怎么把画布重新画出来。

---

### 再看更关键的 components。

生成函数在：web/src/pages/agent/utils\.ts \(line 455\)

函数名是：

```ABAP
buildDslComponentsByGraph(...)
```

它会遍历每一个 node，然后把 node 转成后端执行需要的 component。

最终结构大概是：

```ABAP
components[id] = {
  obj: {
    component_name: operatorName,
    params: params,
  },
  downstream: [...],
  upstream: [...],
  parent_id: x?.parentId,
};
```

换成人话就是：

```ABAP
每个节点会变成一个 component

component 里面保存：
1. 我是什么类型的组件
2. 我的参数是什么
3. 我的下游节点是谁
4. 我的上游节点是谁
5. 我是否属于某个父节点
```

例如你画了这个流程：

```ABAP
Begin -> Retrieval -> Generate
```

前端 DSL 里可能会生成类似结构：

```YAML
components:
  Begin_001:
    component_name: Begin
    params: {...}
    downstream: [Retrieval_001]
    upstream: []

  Retrieval_001:
    component_name: Retrieval
    params:
      kb_ids: [...]
      top_k: 8
      similarity_threshold: 0.2
    downstream: [Generate_001]
    upstream: [Begin_001]

  Generate_001:
    component_name: Generate
    params:
      prompt: ...
      llm_id: ...
    downstream: []
    upstream: [Retrieval_001]
```

这就已经不是“画布”了，而是一个可执行流程图。

---

这里最关键的一点是：**RAGFlow 会把边 edges 转换成每个节点的 upstream/downstream。**

代码在：web/src/pages/agent/utils\.ts \(line 56\)

函数：

```ABAP
buildComponentDownstreamOrUpstream(...)
```

它会扫描所有连线：

```ABAP
如果 edge.source == 当前节点 id
说明这个 edge 指向我的下游

如果 edge.target == 当前节点 id
说明这个 edge 来自我的上游
```

所以：

```ABAP
ReactFlow 的 edges:
  A -> B
  B -> C

会被转换成：

A.downstream = [B]
B.upstream = [A]
B.downstream = [C]
C.upstream = [B]
```

这一步非常关键，因为后端执行时一般不会再从“线条”开始理解流程，而是直接看：

```ABAP
当前组件的下游是谁？
当前组件执行完之后应该走到哪里？
当前组件需要从哪些上游拿输入？
```

---

### 过滤细节

还有一个细节：不是所有画布节点都会进入 components。

代码里有过滤：

```ABAP
const ExcludeOperators = [Operator.Note, Operator.Tool, Operator.Placeholder];
```

也就是说：

```ABAP
Note        不作为执行节点
Placeholder 不作为执行节点
Tool        通常不作为独立主流程节点
```

为什么？

因为：

```ABAP
Note 是注释
Placeholder 是前端占位
Tool 很多时候是 Agent 节点内部挂载的工具
```

它们不一定是主流程里的独立执行单元。

---

### 特殊转换

有些节点还会在生成 DSL 时做特殊转换。

比如：

```YAML
case Operator.Agent:
case Operator.Categorize:
case Operator.Parser:
case Operator.TokenChunker:
case Operator.TitleChunker:
case Operator.Extractor:
case Operator.DataOperations:
case Operator.Begin:
```

这些节点不是简单把 form 原样塞进去，而是会先整理参数。

例如 Categorize 分类节点。

它在画布上可能是这样的：

```ABAP
分类节点
  分支 A -> 节点 1
  分支 B -> 节点 2
```

前端表单里保存的是分类项，但真正后端需要知道的是：

```ABAP
每个分类分支最终连向哪些节点
```

所以代码会把分类节点的连线转换成：

```YAML
category_description:
  类别A:
    description: ...
    examples: [...]
    to: [目标节点id]

  类别B:
    description: ...
    examples: [...]
    to: [目标节点id]
```

对应函数是：

web/src/pages/agent/utils\.ts \(line 140\)

```ABAP
buildCategorize(...)
```

所以分类节点不是只靠表单，它还强依赖连线。

---

### 保存时，调用链是：

web/src/pages/agent/hooks/use\-save\-graph\.ts \(line 17\)

```ABAP
saveGraph()
  -> buildDslData()
  -> setAgent(params)
```

核心代码：

```ABAP
const params = {
  id,
  title: data.title,
  dsl: buildDslData(currentNodes, otherParam),
};

return setAgent(params);
```

也就是说，用户点击保存，或者自动保存时，前端不是只提交 nodes/edges，而是提交完整 DSL：

```ABAP
agent id
agent title
dsl:
  graph
  components
  globals
  variables
  history
  messages
  reference
  retrieval
```

---

这一部分可以总结成一句话：

> 第二部分是 RAGFlow 把前端画布 nodes \+ edges \+ form 转换成 DSL 的过程。其中 graph 用来恢复前端画布，components 用来给后端执行；components 会把每个节点变成一个可执行组件，并把连线转换成 upstream/downstream 关系。
> 
> 

所以从第一部分到第二部分，数据形态发生了这个变化：

```ABAP
画布编辑态
nodes + edges + node.data.form
        ↓
buildDslData()
        ↓
DSL
        ↓
graph: 前端恢复画布
components: 后端执行流程
```
