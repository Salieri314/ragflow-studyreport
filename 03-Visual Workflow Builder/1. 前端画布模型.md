## 第一部分：前端画布模型

### 用户在网页上拖出来的“节点”和“连线”，在前端代码里到底是什么。

RAGFlow 的可视化编排画布，本质上是一个 ReactFlow 画布。

核心代码在：

- web/src/pages/agent/canvas/index\.tsx \(line 304\)

- web/src/pages/agent/store\.ts \(line 55\)

画布上最核心的两个数据就是：

```ABAP
nodes: RAGFlowNodeType[]
edges: Edge[]
```

也就是说，前端看到的 Agent 编排流程，最底层不是神秘的“工作流引擎”，而是两张表一样的东西：

```ABAP
nodes = 所有节点
edges = 所有连线
```

例如你画了一个流程：

```ABAP
Begin -> Retrieval -> LLM -> Answer
```

前端内部大概就是：

```ABAP
nodes:
  Begin 节点
  Retrieval 节点
  LLM 节点
  Answer 节点

edges:
  Begin -> Retrieval
  Retrieval -> LLM
  LLM -> Answer
```

每个 node 不只是一个方块，它还带着这个节点的配置表单。

可以理解成：

```ABAP
一个节点 =
  画布上的位置
  节点类型
  节点 id
  节点显示样式
  节点配置 form
```

比如一个 Retrieval 节点，它不只是显示“知识库检索”几个字，它里面还会保存：

```ABAP
选择了哪些知识库
top_k 是多少
相似度阈值是多少
是否使用 rerank
输出变量叫什么
```

这些配置最终都藏在 node 的 data\.form 里面。

对应的接口入口在：

- web/src/pages/agent/interface\.ts \(line 4\)

这里可以看到一个很关键的结构：

```ABAP
export interface IOperatorForm {
  node?: Operator;
  nodeId?: string;
  form?: any;
}
```

这说明前端把一个节点的“业务配置”抽象成了 form。

所以你可以这样理解：

```ABAP
ReactFlow 负责画图
Zustand store 负责保存图
node.data.form 负责保存每个节点的业务配置
```

用户在画布上拖动节点、连接节点、修改节点配置时，都会更新这个 store。

核心状态在：web/src/pages/agent/store\.ts \(line 55\)

```TypeScript
nodes: RAGFlowNodeType[];
edges: Edge[];
onNodesChange: OnNodesChange;
onEdgesChange: OnEdgesChange;
onConnect: OnConnect;
setNodes: (nodes: RAGFlowNodeType[]) => void;
setEdges: (edges: Edge[]) => void;
updateNodeForm: (nodeId: string, values: any) => void;
```

这几个函数就是画布操作的入口。

比如：

```ABAP
拖动节点       -> onNodesChange
删除连线       -> onEdgesChange
连一条线       -> onConnect
修改表单       -> updateNodeForm
```

其中连线逻辑在：

- web/src/pages/agent/store\.ts \(line 159\)

```ABAP
const newEdges = addEdge(connection, get().edges);
set({ edges: newEdges });
updateFormDataOnConnect(connection);
```

翻译成人话就是：

```ABAP
用户把 A 节点连到 B 节点
前端先把这条线加入 edges
然后根据连接关系，顺手更新相关节点的 form
```

这里为什么要更新 form？

因为有些节点的配置和连线有关。

比如分类节点、工具节点、循环节点，可能需要知道：

```ABAP
我的上游是谁？
我的下游是谁？
某个分支连到了哪个节点？
这个工具调用结果要流向哪里？
```

所以“连线”不只是视觉线条，它也是后续 DSL 生成和执行时的数据依赖关系。

---

### 画布真正渲染的地方在：

- web/src/pages/agent/canvas/index\.tsx \(line 304\)

这里把 store 里的 nodes 和 edges 交给 ReactFlow：

```XML
<ReactFlow
  nodes={nodes}
  edges={edges}
  onNodesChange={onNodesChange}
  onEdgesChange={onEdgesChange}
  onConnect={handleConnect}
  nodeTypes={nodeTypes}
  edgeTypes={edgeTypes}
/>
```

这段代码非常关键。

它说明 RAGFlow 的 Agent Canvas 前端不是自己从零实现画布，而是基于 ReactFlow。

所以第一层模型可以画成这样：

```ABAP
用户操作画布
   ↓
ReactFlow 捕获拖动 / 连线 / 点击
   ↓
useGraphStore 保存 nodes / edges
   ↓
每个 node.data.form 保存节点配置
   ↓
等待下一步转换成 DSL
```

---

### 这一部分最重要的结论是：

前端画布阶段只是“编辑态”。

它保存的是：

```ABAP
节点长什么样
节点在哪里
节点之间怎么连
每个节点配置了什么参数
```

但它还没有真正执行。

真正让它从“图”变成“可执行流程”的，是下一部分：

```ABAP
nodes + edges + form
        ↓
buildDslData()
        ↓
DSL
```

也就是说：

```ABAP
第一部分：画布模型
第二部分：DSL 生成
第三部分：DSL 保存
第四部分：后端加载执行
```

所以你可以先把第一部分理解成：

> RAGFlow 的 Agent 画布前端，本质上是在维护一张由 nodes 和 edges 组成的流程图；每个节点的业务参数存在 node\.data\.form 里。这个阶段只是把用户画出来的流程保存成前端状态，还没有进入真正的执行引擎。
