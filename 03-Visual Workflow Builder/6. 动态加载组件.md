## 第六部分：动态加载组件

RAGFlow 的可插拔组件机制

前面第五部分讲的是：

```ABAP
DSL -> Canvas -> 组件对象
```

第六部分要回答的是：

```ABAP
Canvas 怎么根据一个字符串 "Retrieval" 找到真正的 Retrieval 类？
新增一个组件，为什么主流程不需要大改？
```

核心入口就是：

agent/component/**init**\.py \(line 51\)

```Python
def component_class(class_name):
    for module_name in ["agent.component", "agent.tools", "rag.flow"]:
        try:
            return getattr(importlib.import_module(module_name), class_name)
        except Exception:
            pass
    assert False, f"Can't import {class_name}"
```

这个函数非常关键。

它做的事情就是：

```ABAP
给我一个类名字符串
我去几个模块里找这个类
找到了就返回这个类
```

比如：

```ABAP
component_class("Message")
```

会返回：

```ABAP
agent.component.message.Message
```

再比如：

```ABAP
component_class("Retrieval")
```

会返回：

```ABAP
agent.tools.retrieval.Retrieval
```

再比如 Dataflow 里的：

```ABAP
component_class("Parser")
```

可能会从：

```ABAP
rag.flow
```

里找到对应组件。

---

它查找的顺序是：

```ABAP
1. agent.component
2. agent.tools
3. rag.flow
```

所以 RAGFlow 的组件来源分成三类：

```ABAP
agent.component
  Agent 画布里的基础流程节点

agent.tools
  工具类节点，比如检索、搜索、数据库、外部 API

rag.flow
  Dataflow / 文档处理流水线里的处理组件
```

这就是为什么它可以同时支持：

```ABAP
Agent 编排
工具调用
Dataflow 文档处理
```

因为后端组件查找入口是统一的。

---

但还有一个问题：

```ABAP
agent.component 里面的 Message、Begin、Switch 这些类，是怎么被挂到 agent.component 这个包上的？
```

答案在同一个文件：

agent/component/**init**\.py \(line 20\)

```Java
def _import_submodules():
    for filename in os.listdir(_package_path):
        if filename.startswith("__") or not filename.endswith(".py") or filename.startswith("base"):
            continue
        module_name = filename[:-3]

        module = importlib.import_module(f".{module_name}", package=__name__)
        _extract_classes_from_module(module)
```

这段代码会扫描：

```ABAP
agent/component/
```

下面的所有 \.py 文件。

比如：

```ABAP
begin.py
message.py
switch.py
categorize.py
llm.py
browser.py
...
```

然后逐个 import。

---

导入之后，会调用：

agent/component/**init**\.py \(line 31\)

```ABAP
def _extract_classes_from_module(module):
    for name, obj in inspect.getmembers(module):
        if inspect.isclass(obj) and obj.__module__ == module.__name__ and not name.startswith("_"):
            __all_classes[name] = obj
            globals()[name] = obj
```

这一步的作用是：

```ABAP
从模块里抽取所有公开 class
挂到 agent.component 这个包的全局命名空间
```

举个例子。

agent/component/message\.py 里面有：

agent/component/message\.py \(line 44\)

```ABAP
class MessageParam(ComponentParamBase):
```

以及：

agent/component/message\.py \(line 69\)

```ABAP
class Message(ComponentBase):
    component_name = "Message"
```

当 agent\.component 初始化时，它会自动导入 message\.py，然后把这两个类挂出来：

```ABAP
agent.component.MessageParam
agent.component.Message
```

所以后面：

```ABAP
component_class("Message")
```

就能找到它。

---

这里有一个非常重要的设计约定：

```ABAP
组件类名 = component_name
参数类名 = component_name + "Param"
```

比如：

```ABAP
Begin
BeginParam

Message
MessageParam

Retrieval
RetrievalParam

Switch
SwitchParam

Categorize
CategorizeParam
```

所以第五部分里这段代码才能成立：

agent/canvas\.py \(line 101\)

```ABAP
param = component_class(cpn["obj"]["component_name"] + "Param")()
```

如果 DSL 里写：

```ABAP
"component_name": "Message"
```

Canvas 会找：

```ABAP
MessageParam
```

然后再找：

```ABAP
Message
```

完整流程是：

```ABAP
component_name = "Message"
   ↓
component_class("MessageParam")
   ↓
创建 MessageParam()
   ↓
param.update(params)
   ↓
param.check()
   ↓
component_class("Message")
   ↓
创建 Message(canvas, id, param)
```

这就是插件结构能跑起来的关键。

---

拿 Retrieval 举例。

它不在 agent\.component 里，而是在：

agent/tools/retrieval\.py \(line 37\)

```ABAP
class RetrievalParam(ToolParamBase):
```

以及：

agent/tools/retrieval\.py \(line 86\)

```ABAP
class Retrieval(ToolBase, ABC):
    component_name = "Retrieval"
```

因为 component\_class\(\) 会查：

```ABAP
for module_name in ["agent.component", "agent.tools", "rag.flow"]:
```

所以即使 Retrieval 不在 agent\.component，也能被找到。

这就是多来源兼容。

---

### 这个设计为什么叫“可插拔”？

因为主执行器 Canvas 不需要知道每个组件具体在哪里、怎么实现。

Canvas 只关心：

```ABAP
component_name 是什么？
有没有对应的 Param 类？
有没有对应的组件类？
这个组件能不能 invoke？
```

它不需要写：

```Python
if component_name == "Message":
    from agent.component.message import Message
elif component_name == "Retrieval":
    from agent.tools.retrieval import Retrieval
elif component_name == "Parser":
    from rag.flow.parser.parser import Parser
```

而是统一：

```ABAP
component_class(component_name)
```

这就是它比 if/else 更适合扩展的地方。

如果你要新增一个组件，理论上需要满足几个条件：

```ABAP
1. 在 agent/component 或 agent/tools 或 rag/flow 下新增一个 .py 文件

2. 定义 XxxParam
   继承 ComponentParamBase 或 ToolParamBase

3. 定义 Xxx
   继承 ComponentBase 或 ToolBase

4. 类名和 DSL 里的 component_name 对上

5. 实现 check() 和 invoke() / _invoke()
```

比如你想新增一个组件：

```ABAP
Translate
```

```Python
class TranslateParam(ComponentParamBase):
    def check(self):
        ...

class Translate(ComponentBase):
    component_name = "Translate"

    def _invoke(self, **kwargs):
        ...
```

然后 DSL 写：

```ABAP
{
  "component_name": "Translate",
  "params": {
    "target_language": "en"
  }
}
```

Canvas 加载时就会尝试找：

```ABAP
TranslateParam
Translate
```

只要能找到，就可以实例化。

---

不过这里要注意一点。

RAGFlow 这个机制更准确地说是：

```ABAP
启动时自动扫描 + 运行时按名字查类
```

它不是那种真正意义上的“运行中热插拔”。

也就是说，不是服务运行中你放一个新 \.py 文件进去，它马上自动识别。

它是在 Python 包 import 的时候扫描模块。

---

### 第六部分总结一下：

```ABAP
DSL 写 component_name
   ↓
Canvas.load()
   ↓
component_class(component_name + "Param")
   ↓
创建参数对象
   ↓
component_class(component_name)
   ↓
创建组件对象
   ↓
组件进入 Canvas 执行流程
```

而 component\_class\(\) 的查找范围是：

```ABAP
agent.component
agent.tools
rag.flow
```

所以它能统一加载：

```ABAP
基础流程节点
工具节点
Dataflow 节点
```

一句话总结：

> 第六部分是 RAGFlow 可插拔组件机制的核心。它通过目录扫描、动态 import、类名约定和 component\_class\(\) 查找，把 DSL 里的字符串组件名转换成真实 Python 类，从而让主执行器 Canvas 不需要写死每一种组件。
> 
>
