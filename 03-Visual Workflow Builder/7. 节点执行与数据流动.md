## 第七部分：节点执行与数据流动

```ABAP
DSL -> Canvas -> Component 对象

```


### Canvas\.run\(\) 一开始会把本次请求写入全局变量。

比如：

```Python
for k in kwargs.keys():
    if k in ["query", "user_id", "files", "chat_template_kwargs"] and kwargs[k]:
        self.globals[f"sys.{k}"] = kwargs[k]
```

所以用户问题会变成：

```ABAP
sys.query = 用户输入
```

用户文件会变成：

```ABAP
sys.files = 文件解析结果
```

用户 id 会变成：

```ABAP
sys.user_id = 当前用户
```

这一步就是把外部请求放进 Canvas 的运行上下文。

---

然后它会初始化执行路径：

```ABAP
if not self.path or self.path[-1].lower().find("userfillup") < 0:
    self.path.append("begin")
```

也就是说，正常情况下流程从：

```ABAP
begin
```

开始。

这里的 path 可以理解成一个动态执行队列。

最开始：

```ABAP
path = ["begin"]
```

执行 Begin 之后，会根据 Begin 的下游节点追加新的节点：

```ABAP
path = ["begin", "retrieval_0"]
```

再执行 Retrieval：

```ABAP
path = ["begin", "retrieval_0", "llm_0"]
```

所以 RAGFlow 的执行不是简单递归，也不是先一次性拓扑排序全部节点，而是：

```ABAP
边执行，边往 path 里追加下一个节点
```

---

### 真正执行节点的核心是内部函数：

agent/canvas\.py \(line 438\)

```ABAP
async def _run_batch(f, t):
```

它会执行 path\[f:t\] 这一批节点。

大致逻辑是：

```ABAP
取出当前要执行的节点
准备它的输入
调用 cpn.invoke()
如果是异步组件，就 await cpn.invoke_async()
```

代码里：

```ABAP
cpn = self.get_component_obj(self.path[i])
```

这里拿到的是某个组件对象，比如：

```ABAP
Begin 对象
Retrieval 对象
LLM 对象
Message 对象
Switch 对象
```

然后调用：

```ABAP
task_fn = cpn.invoke
```

或者异步：

```ABAP
await cpn_obj.invoke_async(...)
```

所以每个节点真正干活的地方是组件自己的：

```ABAP
_invoke()
或
_invoke_async()
```

---

那组件的输入从哪里来？

这部分在：

agent/component/base\.py \(line 463\)

```ABAP
def get_input(self, key: str = None)
```

组件执行前，Canvas 会调用：

```ABAP
call_kwargs = cpn.get_input()
```

get\_input\(\) 会看组件参数里有没有变量引用。

比如某个 Message 节点内容里写了：

```ABAP
{{sys.query}}
```

或者：

```ABAP
{{retrieval_0@chunks}}
```

它会识别出来，然后调用 Canvas：

```ABAP
self._canvas.get_variable_value(v)
```

这就进入了变量解析。

---

### 变量主要有三类：

```ABAP
sys.xxx
  系统变量

env.xxx
  用户全局变量

component_id@field
  某个组件的输出
```

比如：

```ABAP
{{sys.query}}
```

代表当前用户问题。

```ABAP
{{retrieval_0@content}}
```

代表 retrieval\_0 这个组件输出里的 content 字段。

```ABAP
{{llm_0@content}}
```

代表 llm\_0 的生成结果。

所以组件之间的数据流，本质不是直接传 Python 参数，而是通过：

```ABAP
组件输出 outputs
+
变量引用 component_id@field
```

来读取。

---

### 每个组件输出在哪里？

在：

agent/component/base\.py \(line 440\)

```ABAP
def output(self, var_nm: str = None)
```

以及：

```ABAP
def set_output(self, key: str, value: Any)
```

一个组件执行时，会把结果写到自己的 outputs 里：

```ABAP
self.set_output("content", result)
```

下游节点如果引用：

```ABAP
{{这个组件id@content}}
```

就能读到这个值。

所以数据流可以理解成：

```ABAP
上游组件 _invoke()
   ↓
set_output("xxx", value)
   ↓
下游组件 get_input()
   ↓
get_variable_value("上游id@xxx")
   ↓
拿到 value
```

这就是 RAGFlow 的组件间传值机制。

---

那执行路径怎么往后走？

每个 component 在 DSL 里都有：

```ABAP
downstream
upstream
```

正常节点执行完后，Canvas 会把下游节点加入 path：

```ABAP
_extend_path(cpn["downstream"])
```

所以普通节点是：

```ABAP
当前节点执行完
   ↓
把 downstream 节点加入 path
   ↓
继续执行
```

例如：

```ABAP
Begin.downstream = [Retrieval]
Retrieval.downstream = [LLM]
LLM.downstream = [Message]
```

运行时就是：

```ABAP
path = [Begin]
执行 Begin
path = [Begin, Retrieval]
执行 Retrieval
path = [Begin, Retrieval, LLM]
执行 LLM
path = [Begin, Retrieval, LLM, Message]
执行 Message
```

---

但是有些节点不会简单走全部 downstream。

比如 Switch。

代码在：

agent/component/switch\.py \(line 53\)

它会根据条件判断下一步去哪：

```ABAP
self.set_output("_next", cond["to"])
```

然后 Canvas 里有特殊处理：

agent/canvas\.py \(line 620\)

```ABAP
elif cpn_obj.component_name.lower() in ["categorize", "switch"]:
    _extend_path(cpn_obj.output("_next"))
```

这说明：

```ABAP
Switch / Categorize 不直接走 cpn["downstream"]
而是走自己输出的 _next
```

也就是说：

```ABAP
普通节点:
  下一步 = downstream

Switch / Categorize:
  下一步 = _next
```

这就是分支逻辑。

---

Categorize 也是类似。

代码在：

agent/component/categorize\.py \(line 98\)

它会让 LLM 判断用户输入属于哪个类别，然后设置：

```ABAP
self.set_output("category_name", max_category)
self.set_output("_next", cpn_ids)
```

所以分类节点执行完后，Canvas 会从 \_next 里拿下一步节点。

比如：

```ABAP
用户问：怎么报销？
Categorize 判断：财务类
_next = [finance_node]
```

于是 path 继续变成：

```ABAP
..., categorize_0, finance_node
```

---

Message 节点比较特殊，因为它可能流式输出。

Canvas 里有专门处理：

agent/canvas\.py \(line 518\)

```ABAP
if cpn_obj.component_name.lower() == "message":
```

如果 Message 的 content 是一个 partial，说明它可以边生成边输出：

```ABAP
stream = cpn_obj.output("content")()
```

然后循环：

```ABAP
yield decorate("message", {"content": m})
```

所以前端能看到答案一点点出来，不是等全部生成完。

这就是：

```ABAP
Message 节点输出 partial / generator
   ↓
Canvas 逐段 yield message
   ↓
SSE 推给前端
```

---

还有一个关键点：Canvas 会发运行事件。

比如节点开始前：

```ABAP
yield decorate("node_started", {...})
```

节点结束后：

```ABAP
yield _node_finished(cpn_obj)
```

整个流程开始：

```ABAP
yield decorate("workflow_started", {...})
```

整个流程结束：

```ABAP
yield decorate("workflow_finished", {...})
```

所以前端看到的“节点亮起来”“节点完成”“消息流式输出”，都来自这些事件。

---

```ABAP
Canvas.run(query)
   ↓
写入 sys.query / sys.files / sys.user_id
   ↓
path.append("begin")
   ↓
while idx < len(path):
   ↓
发送 node_started
   ↓
执行当前批次节点
      cpn.get_input()
      cpn.invoke()
      cpn.set_output(...)
   ↓
发送 node_finished / message
   ↓
决定下一步
      普通节点 -> downstream
      Switch/Categorize -> _next
      Loop/Iteration -> 子节点或回到循环
      UserFillUp -> 暂停等待用户补充输入
   ↓
把下一批节点追加到 path
   ↓
直到没有新节点
   ↓
workflow_finished
```

一句话总结：

> 第七部分的核心是 Canvas\.run\(\)。它把 path 当作动态执行队列，从 begin 开始执行组件；每个组件通过 get\_input\(\) 读取变量和上游输出，通过 set\_output\(\) 写结果；普通节点按 downstream 推进，Switch / Categorize 等特殊节点通过 \_next 改变路径，Message 节点则通过事件流把内容实时返回给前端。
