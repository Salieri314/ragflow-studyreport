## 三\. 后台worker从Redis取任务，并把“小任务信息”恢复为“完整解析上下文”

这一段的核心文件是rag/svr/task\_executor\.py \(line 199\) 和 api/db/services/task\_service\.py \(line 74\)。

前面第二阶段做完之后，Redis队列里大概只有一个很轻的任务消息，比如：

```JSON
{
  "id": "task_001",
  "doc_id": "doc_001",
  "from_page": 0,
  "to_page": 12
}
```

它并不包含完整文档内容，也不包含完整知识库配置。Redis 更像是“任务通知单”：告诉后台 worker，有一个任务可以开始做了。

真正开始处理的是 task\_executor\.py 里的 handle\_task\(\)，位置在 rag/svr/task\_executor\.py \(line 1704\)

它的第一步是：

```ABAP
redis_msg, task = await collect()
```

也就是调用 collect\(\) 去 Redis 拿任务。

collect\(\) 做的事情可以理解成三步。

### 第一步，先确定自己要监听哪些 Redis 队列：

```ABAP
svr_queue_names = settings.get_svr_queue_names(TASK_TYPE)
```

RAGFlow 不是只有一个队列，它可能按优先级、任务类型、服务实例做拆分。所以 worker 会先拿到自己负责消费的队列名。



### 第二步，它不是直接只拿新消息，而是先检查“没确认完成的旧消息”：

```ABAP
REDIS_CONN.get_unacked_iterator(...)
```

这个很重要。

Redis 队列里的任务不是 worker 一拿到就算完成。worker 拿到以后，如果还没处理完就崩了、断电了、进程退出了，这条消息不能丢。所以 RAGFlow 会先检查有没有之前被取走但没有 ack 的任务。

你可以把它理解成快递系统：

- 快递员取走包裹，不代表已经送达

- 只有客户签收了，系统才认为完成

- 如果快递员中途出问题，包裹还得重新派送

这里的 ack\(\) 就是“签收确认”。



### 第三步，如果没有待恢复的旧任务，才从 Redis 正常消费新任务：

```ABAP
redis_msg = REDIS_CONN.queue_consumer(...)
msg = redis_msg.get_message()
```

拿到的 msg 仍然只是 Redis 里的简短任务消息。

真正关键的一步来了：

```ABAP
task = TaskService.get_task(msg["id"])
```

这行在 rag/svr/task\_executor\.py \(line 233\) 附近。

它的意思是：
**Redis 只告诉我任务 ID，我再去 MySQL 里查出完整任务上下文。**

对应代码在 api/db/services/task\_service\.py \(line 74\)。

TaskService\.get\_task\(\) 会把几张表连起来查：

```ABAP
Task
Document
Knowledgebase
Tenant
```

所以原本 Redis 里的小消息：

```JSON
{
  "id": "task_001",
  "doc_id": "doc_001",
  "from_page": 0,
  "to_page": 12
}
```

会被扩展成类似这样的完整任务对象：

```JSON
{
  "id": "task_001",
  "doc_id": "doc_001",
  "from_page": 0,
  "to_page": 12,
  "name": "员工手册.pdf",
  "type": "pdf",
  "location": "xxx.pdf",
  "size": 123456,
  "parser_id": "naive",
  "parser_config": {...},
  "kb_id": "kb_001",
  "tenant_id": "tenant_001",
  "language": "Chinese",
  "embd_id": "bge-large-zh",
  "llm_id": "qwen-plus",
  "img2txt_id": "...",
  "asr_id": "..."
}
```

也就是说，到了这里，后台 worker 才知道：

```ABAP
我要解析哪份文档
文档属于哪个知识库
用哪个 parser
用哪个 embedding 模型
文档原文件在哪里
处理第几页到第几页
租户是谁
知识库语言是什么
是否需要图片理解、语音识别、LLM 等模型能力
```

所以第三阶段的本质是：Redis 负责派活，MySQL 负责补全任务上下文，worker 负责真正干活。

---

### get\_task\(\) 还有两个保护逻辑。

第一个是更新任务状态，写一句：

```ABAP
Task has been received.
```

意思是这个任务已经被后台 worker 收到了。

第二个是重试控制：

```ABAP
if task.retry_count >= 3:
    TaskService.filter_update(..., {"progress_msg": "Task is abandoned."})
    return
```

如果同一个任务已经失败或重试太多次，超过 3 次，RAGFlow 就不再继续死循环处理它，而是标记为 abandoned。

这也是为什么 Redis 任务系统不只是简单“扔进去、拿出来”，它还要配合 MySQL 记录任务生命周期。

---

### 拿到完整 task 后，handle\_task\(\) 会把它放进：

```ABAP
CURRENT_TASKS[task["id"]] = copy.deepcopy(task)
```

这个可以理解成当前 worker 的“正在处理任务表”。后面如果需要查看 worker 当前在跑什么、心跳状态、取消任务，都可以用到这个运行时状态。

接着根据运行模式选择真正的处理函数。默认会走重构后的：

```ABAP
await TaskManager.run_refactored_task(...)
```

旧逻辑则是：

```ABAP
await do_handle_task(task)
```

do\_handle\_task\(\) 开始以后，会继续做几件准备工作：

1. 取出 parser\_id

2. 取出 parser\_config

3. 绑定 embedding 模型

4. 测试 embedding 模型是否可用

5. 初始化知识库索引

6. 判断任务类型

7. 最后才进入真正的解析分块逻辑

其中普通文档最终会走向下一阶段：

```ABAP
chunks = build_chunks(task, progress_callback)
```



### 总结：

现在这一步还不是“解析文档”，而是：worker 从 Redis 接单，然后根据 task\_id 去 MySQL 补全上下文，为正式解析做准备。
