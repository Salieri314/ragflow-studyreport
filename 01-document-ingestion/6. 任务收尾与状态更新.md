## 六\. 任务收尾与状态更新。

索引已经写完了，RAGFlow 怎么把任务标记为完成？前端为什么能看到“解析完成”？失败/取消又是怎么处理的？



它主要有三层状态：

```ABAP
Task 表状态
  单个解析任务的进度

Document 表状态
  整个文档的解析状态

Redis 消息状态
  队列里的任务是否确认完成
```

---

### 第一步：索引写入成功后，更新 chunk/token 统计。

前面第五模块最后，chunk 已经写入 ES / Infinity / OceanBase。接下来会执行 rag/svr/task\_executor\.py \(line 1614\)：

```ABAP
DocumentService.increment_chunk_num(
    task_doc_id,
    task_dataset_id,
    token_count,
    chunk_count,
    0
)
```

这个函数在 api/db/services/document\_service\.py \(line 593\)

它做两件事：

```ABAP
document.token_num += token_num
document.chunk_num += chunk_num
document.process_duration += duration
```

同时还更新知识库：

```ABAP
knowledgebase.token_num += token_num
knowledgebase.chunk_num += chunk_num
```

所以这一步是在 MySQL 里记账：

```ABAP
这个 document 新增了多少 chunk
这个 document 消耗了多少 token
这个 knowledgebase 总共有多少 chunk/token
```

注意：

这里仍然不是把 chunk 正文写进 MySQL。chunk 正文和向量已经在索引库里了。



---

### 第二步：写入索引完成消息

代码在 rag/svr/task\_executor\.py \(line 1655\)

```ABAP
progress_callback(msg="Indexing done ({:.2f}s).".format(timer() - start_ts))
```

progress\_callback 实际上就是前面包装过的 set\_progress\(\)。

set\_progress\(\) 在 rag/svr/task\_executor\.py \(line 165\)。

它会把进度信息写到 task 表里：

```ABAP
d = {"progress_msg": msg}

if prog is not None:
    d["progress"] = prog

TaskService.update_progress(task_id, d)
```

所以当你在前端看到类似：

```ABAP
Indexing done
Embedding chunks
Task done
```

这些很多都来自 Task\.progress\_msg

---

### 第三步：如果有目录 TOC，也插入 TOC chunk。

这一步只针对某些开启了目录提取的情况。

代码在 rag/svr/task\_executor\.py \(line 1657\)：

```ABAP
if toc_thread:
    d = await toc_thread
    if d:
        await _maybe_insert_chunks([d])
        DocumentService.increment_chunk_num(...)
```

意思是：

如果前面异步生成了文档目录，它会把目录也当成一个特殊 chunk 插入索引库。

这个 chunk 会有类似字段：

```ABAP
d["toc_kwd"] = "toc"
d["available_int"] = 0
```

也就是说它不是普通可回答内容，更像是辅助结构信息。



### 第四步：最终把 task 标记为完成。

代码在 rag/svr/task\_executor\.py \(line 1672\)

```ABAP
task_time_cost = timer() - task_start_ts
progress_callback(prog=1.0, msg="Task done ({:.2f}s)".format(task_time_cost))
```

这里的关键是：

```ABAP
prog=1.0
```

也就是把当前 task 的进度写成 100%。

所以对单个 task 来说：

```ABAP
progress = 1.0
progress_msg = Task done (...)
```

这就表示这个任务完成了。

---

### 第五步：document 的状态不是直接在这里简单写死，而是由 task 汇总出来。

这部分在 api/db/services/document\_service\.py \(line 920\) 的 \_sync\_progress\(\)。

它会查这个 document 的所有 task：

```ABAP
tsks = TaskService.query(doc_id=d["id"], order_by=Task.create_time)
```

然后逐个看 task 进度：

```ABAP
if 0 <= t.progress < 1:
    finished = False

if t.progress == -1:
    bad += 1

prg += t.progress if t.progress >= 0 else 0
```

最后算平均值：

```ABAP
prg /= len(tsks)
```

所以如果一个 PDF 被拆成 3 个 task：

```ABAP
task 1: progress = 1.0
task 2: progress = 1.0
task 3: progress = 0.5
```

那 document 进度大概就是：

```ABAP
(1.0 + 1.0 + 0.5) / 3 = 0.833
```

前端看到的文档进度，就是这样从多个 task 汇总出来的。

---

如果所有 task 都完成：

```ABAP
elif finished:
    prg = 1
    status = TaskStatus.DONE.value
```

也就是说 document 会变成：

```ABAP
progress = 1
run = DONE
```

如果所有 task 都结束了，但其中有失败：

```ABAP
if finished and bad:
    prg = -1
    status = TaskStatus.FAIL.value
```

document 会变成：

```ABAP
progress = -1
run = FAIL
```

如果还有 task 没完成：

```ABAP
elif not finished:
    status = TaskStatus.RUNNING.value
```

document 仍然是：

```ABAP
run = RUNNING
```

所以 document 状态是由 task 状态汇总出来的。

---

### 第六步：如果任务失败，写失败状态。

在最外层 handle\_task\(\) 里有异常捕获，位置在 rag/svr/task\_executor\.py \(line 1747\)：

```ABAP
except Exception as e:
    FAILED_TASKS += 1
    set_progress(task_id, prog=-1, msg=f"[Exception]: {err_msg}")
```

这里的关键是：

```ABAP
prog=-1
```

在 RAGFlow 里，progress = \-1 基本表示失败。

而 set\_progress\(\) 里还有一层处理：

```ABAP
if prog is not None and prog < 0:
    msg = "[ERROR]" + msg
```

所以失败信息会写成错误消息，后面 document 汇总时也会被识别成失败。

---

### 第七步：如果任务被取消，清理索引库。

在 rag/svr/task\_executor\.py \(line 1681\) 的 finally 里：

```ABAP
if has_canceled(task_id):
    settings.docStoreConn.delete(
        {"doc_id": task_doc_id},
        search.index_name(task_tenant_id),
        task_dataset_id,
    )
```

意思是：

如果任务执行中被取消，RAGFlow 会尝试把这个 document 已经写入索引库的 chunk 删除掉。

否则可能出现一种不干净的状态：

```ABAP
前端显示文档取消了
但索引库里已经有一部分 chunk
搜索还能搜到它
```

所以取消时要清理索引库。

---

### 第八步：Redis 消息 ack\(\)。

这个是队列层面的最终确认。

代码在 rag/svr/task\_executor\.py \(line 1770\)：

```ABAP
redis_msg.ack()
```

这句发生在 handle\_task\(\) 的最后。

它的意思是：

worker 告诉 Redis：这条消息我处理完了，你可以从未确认队列里移除了。

这里要注意：

ack\(\) 不是业务成功的意思，而是“这条 Redis 消息已经被处理过”。

也就是说：

```ABAP
任务成功 -> set_progress(1.0) -> ack
任务失败 -> set_progress(-1) -> ack
任务取消 -> 清理/标记取消 -> ack
```

如果 worker 在执行过程中崩了，还没来得及 ack\(\)，那这条消息会留在 unacked 队列里，后面可以被重新恢复处理。

---

所以第六大模块完整流程是：

```YAML
索引写入成功
  |
  v
更新 document / knowledgebase 的 chunk_num、token_num
  |
  v
写入 task progress_msg: Indexing done
  |
  v
写入 task progress = 1.0: Task done
  |
  v
DocumentService._sync_progress 汇总所有 task
  |
  v
document.run = DONE / FAIL / RUNNING
document.progress = 1 / -1 / 平均进度
  |
  v
前端查询 document 状态并展示
  |
  v
Redis ack，队列消息确认完成
```

一句话总结：

第六大模块就是把“后台实际干完了”转换成“数据库状态可见、前端能看到完成、Redis 队列也确认结束”的闭环。
