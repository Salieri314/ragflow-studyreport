## 四\. build\_chunks\(\) 如何选解析器、读原始文件，并把文档交给 chunker

这一段入口在 rag/svr/task\_executor\.py \(line 270\)：

```ABAP
async def build_chunks(task, progress_callback):
```

它接收的是上一阶段准备好的完整 task。这个 task 已经包含：

```JSON
{
  "doc_id": "doc_001",
  "parser_id": "naive",
  "parser_config": {...},
  "from_page": 0,
  "to_page": 12,
  "kb_id": "kb_001",
  "tenant_id": "tenant_001",
  "language": "Chinese",
  "name": "员工手册.pdf",
  "size": 123456
}
```

所以这一阶段的核心问题是：我拿到了一个文档任务，现在到底该用哪个解析方法？原始文件在哪里？怎么把它交给解析器？

### 第一步，先检查文件大小。

代码在 rag/svr/task\_executor\.py \(line 271\)：

```ABAP
if task["size"] > settings.DOC_MAXIMUM_SIZE:
    set_progress(...)
    return []
```

意思是如果文件超过系统允许的最大大小，就直接失败，不进入解析。

这一步是保护后台 worker 的。因为解析 PDF、OCR、图片识别都可能很重，如果一个文件特别大，可能会拖垮整个任务执行器。

### 第二步，根据 parser\_id 找对应的 chunker。

代码在 rag/svr/task\_executor\.py \(line 279\)：

```ABAP
chunker = FACTORY[task["parser_id"].lower()]
```

这里非常关键。

parser\_id 是前面文档上传时已经决定好的，比如：

```ABAP
naive
paper
book
presentation
manual
qa
table
picture
audio
email
```

而 FACTORY 是一个映射表，在 rag/svr/task\_executor\.py \(line 115\)

```YAML
FACTORY = {
    "general": naive,
    ParserType.NAIVE.value: naive,
    ParserType.PAPER.value: paper,
    ParserType.BOOK.value: book,
    ParserType.PRESENTATION.value: presentation,
    ParserType.MANUAL.value: manual,
    ParserType.LAWS.value: laws,
    ParserType.QA.value: qa,
    ParserType.TABLE.value: table,
    ParserType.RESUME.value: resume,
    ParserType.PICTURE.value: picture,
    ParserType.ONE.value: one,
    ParserType.AUDIO.value: audio,
    ParserType.EMAIL.value: email,
    ParserType.KG.value: naive,
    ParserType.TAG.value: tag
}
```

也就是说：

```ABAP
parser_id = naive        -> rag/app/naive.py
parser_id = table        -> rag/app/table.py
parser_id = picture      -> rag/app/picture.py
parser_id = audio        -> rag/app/audio.py
parser_id = presentation -> rag/app/presentation.py
parser_id = email        -> rag/app/email.py
```

所以 RAGFlow 不是在 build\_chunks\(\) 里写一堆 if/else 判断：

```ABAP
如果是 PDF 怎么办
如果是 Excel 怎么办
如果是图片怎么办
```

而是用 parser\_id 找到对应模块，然后统一调用这个模块的：

```ABAP
chunker.chunk(...)
```

这就是它的插件式解析结构。

其实也就是前面的流程以及帮他把parse\_id的判断做了，这里只需要执行就行。

### 第三步，去对象存储里拿原始文件。

代码在 rag/svr/task\_executor\.py \(line 282\)：

```ABAP
bucket, name = File2DocumentService.get_storage_address(doc_id=task["doc_id"])
binary = await get_storage_binary(bucket, name)
```

这一步对应我们之前说的存储分工：

```ABAP
MySQL 里存 document / file / file2document 元数据
MinIO / S3 / OSS 里存真正的原始文件
```

所以 worker 不是从 MySQL 里读 PDF 内容。MySQL 只告诉它：

```ABAP
这个 doc_id 对应的文件放在哪个 bucket、哪个 object name
```

然后它去 MinIO/S3/OSS 里把原始文件读成二进制：

```ABAP
binary
```

你可以把 binary 理解成：

```ABAP
员工手册.pdf 这个文件本身的字节内容
```

还没有解析，只是把文件拿到了内存里。

### 第四步，合并解析配置。

代码在 rag/svr/task\_executor\.py \(line 302\)：

```ABAP
parser_config_for_chunk = merge_table_parser_config_from_kb(task)
```

这个主要是为了表格类文档。因为有些解析配置可能存在 document 上，有些配置可能存在 knowledgebase 上。

比如表格解析时，用户可能指定：

```ABAP
哪一列是问题
哪一列是答案
哪一列是分类
```

这些配置不一定只属于单个文档，也可能属于整个知识库。所以这里会把相关配置合并成一个最终版本：

```ABAP
parser_config_for_chunk
```

后面真正解析时，用的是合并后的配置，而不是简单只用 task\["parser\_config"\]。

### 第五步，记录本次 chunk 配置。

代码在 rag/svr/task\_executor\.py \(line 312\)：

```JSON
chunk_config = {
    "parser_id": task["parser_id"],
    "chunk_token_num": parser_config_for_chunk.get("chunk_token_num", 128),
    "overlapped_percent": ...,
    "delimiter": ...,
    "from_page": task["from_page"],
    "to_page": task["to_page"],
    "language": task["language"],
    "layout_recognizer": parser_config_for_chunk.get("layout_recognizer"),
}
```

这一步不是解析本身，而是把“这次怎么切”记录下来。

例如一份员工手册可能配置为：

```ABAP
每块 512 token
相邻块重叠 10%
按换行/句号/问号切分
只解析第 0 到 12 页
语言是中文
使用 deepdoc 版面识别
```

这些都会影响最终 chunk 的形状。

### 第六步，真正调用 chunker，把原始文件解析并切成初始 chunks

代码在 rag/svr/task\_executor\.py \(line 327\)：

```SQL
async with chunk_limiter:
    cks = await thread_pool_exec(
        chunker.chunk,
        task["name"],
        binary=binary,
        from_page=task["from_page"],
        to_page=task["to_page"],
        lang=task_language,
        callback=progress_callback,
        kb_id=task["kb_id"],
        parser_config=parser_config_for_chunk,
        tenant_id=task["tenant_id"],
    )
```

这里有几个点很重要。

chunk\_limiter 是并发限制。因为解析文档很吃 CPU、内存、OCR 资源，所以不能无限并发。

thread\_pool\_exec 是把同步的解析函数放进线程池里执行。因为很多 parser 不是 async 写法，直接跑会阻塞主事件循环。

真正被调用的是：

```ABAP
chunker.chunk(...)
```

如果 parser\_id = naive，就是调用：

```ABAP
rag/app/naive.py 里的 chunk()
```

如果 parser\_id = table，就是调用：

```ABAP
rag/app/table.py 里的 chunk()
```

如果 parser\_id = picture

```ABAP
rag/app/picture.py 里的 chunk()
```

所以这一句才是真正进入“不同解析方法”的分叉点。

---

返回值 cks 是什么？

```ABAP
cks = ...
```

cks 可以理解成 chunk 列表，也就是文档被拆出来的一块块内容。

大概长这样：

```JSON
[
  {
    "content_with_weight": "第一章 总则。为了规范员工行为...",
    "page_num_int": 1,
    "position_int": [[1, 100, 200, 300, 400]],
    "image": "...可能有截图..."
  },
  {
    "content_with_weight": "第二章 考勤制度。员工应当...",
    "page_num_int": 2,
    "position_int": [[2, 120, 220, 320, 420]]
  }
]
```

注意，这时候还没有进入向量化，也还没有写入 ES / Infinity。

这里只是完成了：

```ABAP
原始文件 -> parser/chunker -> chunks
```

所以第四阶段可以画成这样：

```YAML
完整 task
  |
  v
检查文件大小
  |
  v
根据 parser_id 从 FACTORY 选择 chunker
  |
  v
通过 doc_id 找到对象存储地址
  |
  v
从 MinIO / S3 / OSS 读取原始文件 binary
  |
  v
合并 parser_config
  |
  v
调用 chunker.chunk(...)
  |
  v
得到 chunks 列表 cks
```

build\_chunks\(\) 本身不是具体解析器，它是解析调度入口；真正的解析逻辑在 rag/app/naive\.py、rag/app/table\.py、rag/app/picture\.py 等模块里。DeepDOC 通常是在这些 chunker 内部被调用，而不是在最外层直接统一调用。

---

真正调用 chunker\.chunk\(\.\.\.\) 之后发生了什么？

入口还是 rag/svr/task\_executor\.py \(line 327\)：

```SQL
cks = await thread_pool_exec(
    chunker.chunk,
    task["name"],
    binary=binary,
    from_page=task["from_page"],
    to_page=task["to_page"],
    lang=task_language,
    callback=progress_callback,
    kb_id=task["kb_id"],
    parser_config=parser_config_for_chunk,
    tenant_id=task["tenant_id"],
)
```

这行可以翻译成一句话：

把原始文件 binary、页码范围、语言、解析配置等参数，交给对应的 chunker，让它返回一组 chunks。

以最常见的 naive 为例。如果前面得到的是：

```ABAP
chunker = naive
```

那么这里实际调用的就是：

```ABAP
rag/app/naive.py 里的 chunk(...)
```

也就是 rag/app/naive\.py \(line 792\)。

这个 chunk\(\) 才会继续判断具体文件类型：

```ABAP
PDF 怎么处理
DOCX 怎么处理
TXT 怎么处理
Markdown 怎么处理
Excel 怎么处理
HTML 怎么处理
JSON 怎么处理
```

所以这里要注意一件事：

parser\_id 决定“解析策略”，但进入某个策略以后，里面还会根据文件扩展名继续细分处理。

也就是说，判断不是只有一层，而是两层。

第一层：这个文档走哪个 chunker？

例如：naive / table / paper / presentation / picture / audio

第二层：在这个 chunker 里面，具体文件格式是什么？

例如在 naive 里面：

```ABAP
.pdf
.docx
.txt
.md
.xlsx
.html
.json
```

---

返回值 cks 就是 chunks。

每个 chunk 大概像这样：

```ABAP
{
  "content_with_weight": "员工应当遵守公司考勤制度，每日按规定时间打卡...",
  "page_num_int": 3,
  "position_int": [[3, 120, 180, 500, 260]]
}
```

如果是带图片或截图的 PDF，可能还会有：

```ABAP
{
  "content_with_weight": "图 2：审批流程示意图...",
  "image": "...",
  "page_num_int": 5
}
```

如果是表格，可能会有更结构化的字段。

---

所以这一阶段完成的是：

```ABAP
原始文件 binary
   |
   v
对应 chunker.chunk(...)
   |
   v
解析文本 / 表格 / 图片 / 页码 / 位置信息
   |
   v
切成 chunks
   |
   v
返回 cks
```

但注意，到这里为止：

```ABAP
还没有 embedding
还没有向量
还没有写入 ES / Infinity
```

这一步只是把“文件”变成“可检索的文本块候选”。



### 第七步，cks 返回以后，RAGFlow 如何整理成最终 docs。

上一小步结束时，chunker\.chunk\(\.\.\.\) 返回了：cks

它可以理解成“解析器吐出来的原始 chunks”。但这些 chunks 还不一定能直接写入索引库，因为它们可能还缺少：

```ABAP
doc_id
kb_id
chunk id
创建时间
图片资源地址
关键词
问题
标签
metadata
```

所以 build\_chunks\(\) 后半段做的事情就是：

把 cks 包装、补字段、补资源、补增强信息，最后变成可以进入 embedding / 索引写入阶段的 docs。

代码从 rag/svr/task\_executor\.py \(line 370\) 开始。

首先它创建一个空列表：

```ABAP
docs = []
```

这个 docs 就是最终要返回的 chunk 文档列表。

然后创建一个基础对象：

```ABAP
doc = {
    "doc_id": task["doc_id"],
    "kb_id": str(task["kb_id"])
}
```

这一步很重要。

因为 chunker\.chunk\(\.\.\.\) 返回的每个 chunk 可能只知道：

```ABAP
这一块的文本是什么
在哪一页
位置在哪里
有没有图片
```

但它不一定知道：

```ABAP
我属于哪个 document
我属于哪个 knowledge base
```

所以这里统一补上：

```ABAP
doc_id = 当前文档 ID
kb_id = 当前知识库 ID
```

也就是说，原始 chunk 可能长这样：

```ABAP
{
  "content_with_weight": "员工应当遵守公司考勤制度...",
  "page_num_int": 3
}
```

经过包装后，会变成：

```JSON
{
  "doc_id": "doc_001",
  "kb_id": "kb_001",
  "content_with_weight": "员工应当遵守公司考勤制度...",
  "page_num_int": 3
}

```

然后它定义了一个内部函数：

```ABAP
async def upload_to_minio(document, chunk):
```

这个名字叫 upload\_to\_minio，但它不只是上传图片，它还负责把单个 chunk 整理成最终 doc。

核心逻辑在 rag/svr/task\_executor\.py \(line 382\)：

```ABAP
d = copy.deepcopy(document)
d.update(chunk)
```

这两行的意思是：

1. 先复制基础信息：

```ABAP
{
  "doc_id": "doc_001",
  "kb_id": "kb_001"
}
```

2. 再把 chunk 自己的内容合并进去：

```ABAP
{
  "content_with_weight": "员工应当遵守公司考勤制度...",
  "page_num_int": 3
}
```

合并后：

```JSON
{
  "doc_id": "doc_001",
  "kb_id": "kb_001",
  "content_with_weight": "员工应当遵守公司考勤制度...",
  "page_num_int": 3
}
```

这就是从 ck 变成 d 的过程。

---

接着它给每个 chunk 生成一个唯一 ID：

```ABAP
d["id"] = xxhash.xxh64(
    (chunk["content_with_weight"] + str(d["doc_id"])).encode("utf-8", "surrogatepass")
).hexdigest()
```

这个 ID 不是随机生成的，而是根据：

```ABAP
chunk 文本内容 + doc_id
```

算出来的 hash。

举个例子：

```ABAP
content = "员工应当遵守公司考勤制度..."
doc_id = "doc_001"
```

它会把这两个拼起来，然后算出一个类似这样的 ID：

```ABAP
9f2a31c8e7...
```

为什么要这样做？

因为同一份文档里，同一段内容应该对应稳定的 chunk ID。

而加上 doc\_id 是为了避免不同文档里刚好有同一句话，结果 chunk ID 冲突。

也就是说：

```ABAP
同样的文本 + 同一个 doc_id -> 同一个 chunk id
同样的文本 + 不同 doc_id -> 不同 chunk id
```

---

然后补创建时间：

```ABAP
d["create_time"] = ...
d["create_timestamp_flt"] = ...
```

这两个字段后面写入 ES / Infinity 时会用到，方便排序、过滤、展示。

---

接下来处理图片。

代码在 rag/svr/task\_executor\.py \(line 389\)：

```ABAP
if d.get("img_id"):
    docs.append(d)
    return
```

意思是：如果 chunk 已经有 img\_id，说明图片已经处理过了，直接加入 docs。

然后：

```ABAP
if not d.get("image"):
    _ = d.pop("image", None)
    d["img_id"] = ""
    docs.append(d)
    return
```

意思是：如果这个 chunk 没有图片，那就把 image 字段清掉，并设置：

```ABAP
img_id = ""
```

然后加入 docs。

如果这个 chunk 有图片：

```ABAP
await image2id(...)
docs.append(d)
```

这一步会把 chunk 里的图片内容存到对象存储里，然后给它生成一个 img\_id。

所以图片处理可以理解成：

```ABAP
chunk 没图片
  -> img_id = ""

chunk 已有 img_id
  -> 直接保留

chunk 有 image 原始内容
  -> 上传到 MinIO/S3/OSS
  -> 转成 img_id
```

这就是为什么 RAGFlow 的 chunk 不一定只有文本，它也可能带图片引用。

---

然后它对所有 chunks 并发执行这个整理函数：

```ABAP
for ck in cks:
    tasks.append(asyncio.create_task(upload_to_minio(doc, ck)))

await asyncio.gather(*tasks)
```

也就是说，假设 cks 有 100 个 chunk，它会把这 100 个 chunk 都整理成 docs 里的对象。

结束后，docs 大概是这样：

```JSON
[
  {
    "id": "chunk_hash_001",
    "doc_id": "doc_001",
    "kb_id": "kb_001",
    "content_with_weight": "员工应当遵守公司考勤制度...",
    "page_num_int": 3,
    "img_id": "",
    "create_time": "2026-06-09 14:30:00",
    "create_timestamp_flt": 1780986600.12
  },
  {
    "id": "chunk_hash_002",
    "doc_id": "doc_001",
    "kb_id": "kb_001",
    "content_with_weight": "审批流程如下图所示...",
    "page_num_int": 5,
    "img_id": "image_xxx",
    "create_time": "2026-06-09 14:30:00"
  }
]
```

这一步完成以后，chunk 已经从“解析器内部结果”变成了“RAGFlow 索引文档块”。

### 可选增强

第一类：自动关键词。
