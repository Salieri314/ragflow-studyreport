然后写入 ES / Infinity / OceanBase。

### 第一步：先绑定 embedding 模型。

代码在 rag/svr/task\_executor\.py \(line 1405\)：

```ABAP

    embd_model_config = get_model_config_by_type_and_name(
```

比如可能是：

```ABAP

text-embedding-3-large
BAAI/bge-m3
```

这里不会直接写死模型，而是根据租户配置和知识库配置动态加载。

然后它会测试一下模型能不能用：

```ABAP

```

这行很有意思。它先拿 "ok" 做一次试编码。目的不是为了处理文档，而是为了确认：

```ABAP
embedding 模型能调用成功
向量维度是多少
```

这个 vector\_size 后面创建索引时要用。

### 第二步：初始化知识库索引。

代码在 rag/svr/task\_executor\.py \(line 1419\)：

```ABAP

```

对应函数在 rag/svr/task\_executor\.py \(line 689\)：

这一步是在检索存储里创建或确认索引存在。

这里要注意：

索引不是按文件名建的，也不是每个文件一个索引。

它通常是：

```ABAP
tenant 维度的 index name
+ kb_id 维度的知识库分区/集合
```

大致可以理解成：

```ABAP
这个租户有一个索引空间
这个知识库在里面有自己的数据范围
```

vector\_size 很关键。因为向量字段必须知道维度，比如：

```ABAP
q_1024_vec
q_1536_vec
```

不同 embedding 模型维度不同，索引结构也要匹配。

### 第三步：调用 embedding\(chunks, embedding\_model, \.\.\.\)

代码在 rag/svr/task\_executor\.py \(line 1557\)：

```ABAP
token_count, vector_size = await embedding(
    chunks,
    embedding_model,
    task_parser_config,
    progress_callback
)
```

真正的 embedding 函数在 rag/svr/task\_executor\.py \(line 696\)

它先准备两类文本：

```ABAP
tts = []
cnts = []
```

tts 是标题文本：

```ABAP
tts.append(d.get("docnm_kwd", "Title"))
```

也就是文档名或标题。

cnts 是 chunk 内容：

```ABAP
c = "\n".join(d.get("question_kwd", []))
if not c:
    c = d["content_with_weight"]
```

这里有个重点：

如果前面开启了自动问题生成，并且 chunk 里有 question\_kwd，RAGFlow 会优先拿“问题”去做 embedding；如果没有问题，才拿原文内容做 embedding。

为什么？

因为用户检索时通常是提问，不是复述原文。

比如原文是：

```ABAP
员工每日上午 9 点前完成打卡。
```

用户问：

```ABAP
上班几点前要打卡？
```

如果提前生成了问题：

```ABAP
员工几点前需要打卡？
```

它可能比原文更接近用户查询。

所以这里不是简单“永远把原文 embed”，而是：

```ABAP
有自动问题 -> embed 问题
没有自动问题 -> embed 原文
```

### 第四步：批量调用 embedding 模型。

代码在 rag/svr/task\_executor\.py \(line 722\):

```ABAP
for i in range(0, len(cnts), settings.EMBEDDING_BATCH_SIZE):
    async with embed_limiter:
        vts, c = await thread_pool_exec(
            batch_encode,
            cnts[i: i + settings.EMBEDDING_BATCH_SIZE]
        )
```

这里不是一条一条算，而是按 batch 算。

比如有 1000 个 chunks，系统可能每 16 个或 32 个一批送入 embedding 模型。

embed\_limiter 是并发限制，防止同时太多 embedding 请求打爆模型服务。

batch\_encode\(\) 里面还有一层：

```ABAP
truncate(c, mdl.max_length - 10)
```

意思是如果 chunk 太长，会截断到 embedding 模型允许的最大长度附近。

### 第五步：把“标题向量”和“正文向量”加权融合。

代码在 rag/svr/task\_executor\.py \(line 730\)

```ABAP
filename_embd_weight = parser_config.get("filename_embd_weight", 0.1)
title_w = float(filename_embd_weight)

vects = title_w * tts + (1 - title_w) * cnts
```

这个逻辑很重要。

RAGFlow 不一定只用正文向量，它会把标题/文件名的向量也混进去一点。

默认权重是：

```ABAP
标题/文件名 10%
正文内容 90%
```

为什么这么做？

因为文件名有时候很有信息量。

比如 chunk 内容是：

```ABAP
第一条 本制度适用于全体员工。
```

单看这句话，不知道是什么制度。

但文件名是：

```ABAP
考勤管理制度.pdf
```

那这个 chunk 就明显和“考勤”有关。

所以最终向量大概是：

```ABAP
最终向量 = 0.1 * 文件名向量 + 0.9 * chunk 内容向量
```

这能让检索时更容易命中文档主题。


### 总结

每一个 chunk 的主体内容保存到索引库；MySQL 主要保存文档级状态和统计，不逐条保存每个 chunk 的完整内容。

也就是：

```ABAP
chunk 本体
  -> ES / Infinity / OceanBase

chunk 数量、token 数、解析状态、进度
  -> MySQL 的 document / task 等表
```

比如一个 chunk：

```JSON
{
  "id": "chunk_001",
  "doc_id": "doc_001",
  "kb_id": "kb_001",
  "content_with_weight": "员工应当遵守公司考勤制度...",
  "q_1024_vec": [0.01, -0.02, ...],
  "page_num_int": 3
}
```

这个完整 chunk 会写入：

```ABAP
ES / Infinity / OceanBase
```

因为它要被检索：

```ABAP
全文检索
向量检索
metadata 过滤
页码定位
图片定位
```

---

MySQL 不会把每个 chunk 的正文和向量都存一份。

MySQL 主要更新类似这些信息：

```ABAP
document.chunk_num
document.token_num
document.progress
document.run
task.progress
task.progress_msg
document metadata
```

例如代码里这一步：

```ABAP
DocumentService.increment_chunk_num(
    task_doc_id,
    task_dataset_id,
    token_count,
    chunk_count,
    0
)
```

意思是告诉 MySQL：

```ABAP
这个 document 新增了多少 chunk
这些 chunk 总共有多少 token
文档解析状态应该更新
```

但不是把每个 chunk 内容插入 MySQL。

---

所以最终可以这样理解：

```ABAP
MySQL:
  管“文档是谁、属于哪个知识库、解析状态、任务进度、chunk 数、token 数”

索引库 ES / Infinity / OceanBase:
  管“每个 chunk 的正文、向量、页码、位置、关键词、metadata”

对象存储 MinIO / S3 / OSS:
  管“原始文件、chunk 图片”
```

也就是说：

MySQL 管账本，索引库管可检索内容，对象存储管原始文件和图片。
