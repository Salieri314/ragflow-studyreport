## 二\. 会话与配置加载

这一步还没有真正检索，也还没有调用 LLM。它做的是：

先弄清楚“谁在问、问的是哪个 Chat、属于哪个 Session、应该用哪些知识库和模型配置”。

入口还是 api/apps/restful\_apis/chat\_api\.py \(line 1126\)：

```Python
@manager.route("/chat/completions", methods=["POST"])
@login_required
async def session_completion(chat_id_in_arg=""):
```

### 第一步是读取 JSON：

```ABAP
req = await get_request_json()
```

然后把请求里的 message 格式统一一下：

```ABAP
normalized, error = _normalize_completion_messages(req)
request_messages, request_msg = normalized
```

这里是为了兼容不同输入形式。

有些请求可能是：

```ABAP
{
  "question": "员工迟到怎么处理？"
}
```

有些可能是 OpenAI 风格：

```ABAP
{
  "messages": [
    {"role": "user", "content": "员工迟到怎么处理？"}
  ]
}
```

RAGFlow 会把它们整理成内部统一格式：

```ABAP
request_msg
```

后面统一按 message list 来处理。

---

### 然后取几个关键 ID：

```Java
chat_id = req.pop("chat_id", "") or ""
chat_id = chat_id or chat_id_in_arg

session_id = req.pop("session_id", "") or req.pop("conversation_id", "") or ""

chat_model_id = req.pop("llm_id", "")
```

这几个字段含义分别是：

```ABAP
chat_id:
  当前使用的是哪个 Chat 助手 / dialog 配置

session_id:
  当前是哪一段对话会话

llm_id:
  本次请求是否临时指定某个 LLM
```

比如用户在某个 Chat 页面问：

```ABAP
员工迟到怎么处理？
```

后端需要知道：

```ABAP
这是哪个 Chat？
这个 Chat 绑定了哪些知识库？
这个 Chat 用哪个模型？
这是新会话还是旧会话？
```

这些都不是靠问题文本本身能判断出来的，要从 MySQL 里读配置。



然后它读取请求里的模型生成参数：

```ABAP
chat_model_config = {}

for model_config in [
    "temperature",
    "top_p",
    "frequency_penalty",
    "presence_penalty",
    "max_tokens"
]:
    config = req.get(model_config)
    if config:
        chat_model_config[model_config] = config
```

这些是 LLM 生成参数。

比如：

```ABAP
temperature:
  回答随机性

top_p:
  采样范围

max_tokens:
  最大生成长度
```

如果请求里带了这些参数，就会覆盖或补充 Chat 默认模型配置。

---

### 接下来进入主逻辑。

如果有 chat\_id，说明这是一个已创建的 RAG Chat。

代码在 api/apps/restful\_apis/chat\_api\.py \(line 1154\)：

```ABAP
if chat_id:
    if not await _ensure_owned_chat(chat_id):
        return ...
```

\_ensure\_owned\_chat\(\) 在 api/apps/restful\_apis/chat\_api\.py \(line 133\)：

```ABAP
return await thread_pool_exec(
    DialogService.query,
    tenant_id=current_user.id,
    id=chat_id,
    status=StatusEnum.VALID.value
)
```

意思是：先确认这个 Chat 属于当前登录用户，并且状态是有效的。

所以这里是权限检查。

---

### 权限通过以后，读取 Chat 配置：

```ABAP
e, dia = await thread_pool_exec(DialogService.get_by_id, chat_id)
```

这里的 dia 可以理解成 dialog/chat 配置对象。

它里面会包含这些核心配置：

```ABAP
tenant_id
llm_id
llm_setting
prompt_config
kb_ids
top_n
top_k
rerank_id
similarity_threshold
vector_similarity_weight
meta_data_filter
```

这些字段来自 Dialog 表，也就是 RAGFlow 里 Chat 助手的配置。

比如一个 Chat 可能配置为：

```JSON
{
  "name": "HR 助手",
  "kb_ids": ["kb_employee_handbook"],
  "llm_id": "qwen-plus",
  "top_k": 1024,
  "top_n": 6,
  "similarity_threshold": 0.2,
  "vector_similarity_weight": 0.3,
  "rerank_id": "bge-reranker",
  "prompt_config": {
    "system": "你是 HR 助手，请基于知识库回答问题。{knowledge}",
    "prologue": "你好，我是 HR 助手。",
    "quote": true
  }
}
```

所以第二模块最核心的一步就是：

根据 chat\_id 从 MySQL 取出 dia，后续检索和生成都按它的配置来。

---

### 然后处理 session\_id。

如果请求带了 session\_id：

```ABAP
e, conv = await thread_pool_exec(ConversationService.get_by_id, session_id)
```

这表示继续一段已有会话。

接着检查：

```ABAP
if conv.dialog_id != chat_id:
    return get_data_error_result(message="Session does not belong to this chat!")
```

这个检查很重要。

它防止用户拿 A Chat 的 session\_id 去套 B Chat。

也就是说：session 必须属于当前 chat

---

如果没有 session\_id，就创建新会话：

```ABAP
conv = await _create_session_for_completion(chat_id, dia, current_user.id)
session_id = conv.id
```

创建函数在 api/apps/restful\_apis/chat\_api\.py \(line 157\)：

```JSON
conv = {
    "id": get_uuid(),
    "dialog_id": chat_id,
    "name": "New session",
    "message": [
        {
          "role": "assistant",
          "content": dialog.prompt_config.get("prologue", "")
        }
    ],
    "user_id": user_id,
    "reference": [],
}
```

这里可以看到，新会话一开始会放一条 assistant 消息：

```ABAP
prologue
```

也就是 Chat 配置里的开场白。

比如：

```ABAP
你好，我是 HR 助手，有什么可以帮你？
```

然后写入 MySQL：

```ABAP
ConversationService.save(**conv)
```

所以 conversation/session 是持久化在 MySQL 里的。

---

### 然后它整理对话历史。

代码在 api/apps/restful\_apis/chat\_api\.py \(line 1174\)

```Go
if pass_all_history_messages:
    conv.message = deepcopy(request_messages)
    msg = request_msg
else:
    if not conv.message:
        conv.message = []
    conv.message.append(deepcopy(request_msg[-1]))
    msg = []
    for m in conv.message:
        if m["role"] == "system":
            continue
        if m["role"] == "assistant" and not msg:
            continue
        msg.append(m)
```

这段有点绕，翻译成人话是：

如果请求要求“传入完整历史”：

```ABAP
直接使用请求里的 messages 作为历史
```

否则：

```ABAP
把本轮用户问题追加到已有 conv.message
然后从 conv.message 里整理出后面要送给 async_chat 的 msg
```

它会跳过：

```ABAP
system 消息
开头多余的 assistant 消息
```

最终得到的 msg 是后面真正参与 RAG/LLM 的对话上下文。

---

### 然后为本轮会话准备引用槽位：

```ABAP
if conv is not None:
    if not conv.reference:
        conv.reference = []
    conv.reference = [r for r in conv.reference if r]
    conv.reference.append({"chunks": [], "doc_aggs": []})
```

这个 reference 是后面存引用用的。

也就是说，本轮回答会有一个新的引用位置：

```ABAP
{
  "chunks": [],
  "doc_aggs": []
}
```

等后面检索到了 chunks，会把引用信息填进去。

所以 conversation 里不只是保存对话文本，还会保存每轮回答对应的引用来源。

---

### 如果本次请求临时指定了 llm\_id：

```ABAP
if chat_model_id:
    if not await thread_pool_exec(
        TenantLLMService.get_api_key,
        tenant_id=dia.tenant_id,
        model_name=chat_model_id
    ):
        return ...
    dia.llm_id = chat_model_id
    dia.llm_setting = chat_model_config
```

这表示：

用户可以在请求级别覆盖 Chat 默认模型，但前提是当前租户真的配置了这个模型。

如果没有 API key 或模型不可用，就拒绝。

如果没有 chat\_id，它会走另一个分支：

```ABAP
dia = _build_default_completion_dialog()
```

这个是“直接 completion”模式，不绑定知识库：

```ABAP
api/apps/restful_apis/chat_api.py (line 140)
```

```ABAP
kb_ids=[]
top_n=6
top_k=1024
rerank_id=""
similarity_threshold=0.1
vector_similarity_weight=0.3
```

注意这里：

```ABAP
kb_ids=[]
```

说明它不是典型知识库 RAG Chat。我们的报告主线先不展开这个。

---

所以第 2 大模块可以总结成：

```YAML
请求 JSON
  |
  v
统一消息格式
  |
  v
取 chat_id / session_id / llm_id
  |
  v
校验 Chat 权限
  |
  v
读取 Dialog 配置 dia
  |
  v
读取或创建 Conversation conv
  |
  v
整理历史消息 msg
  |
  v
准备 reference 空槽
  |
  v
可选覆盖 LLM 配置
  |
  v
进入 async_chat(dia, msg, ...)
```
