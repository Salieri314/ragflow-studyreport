## 七\. 返回前端与保存会话

```ABAP
LLM 已经生成答案了，
RAGFlow 怎么把它返回给前端？
怎么保存 conversation.message？
怎么保存 reference？
流式返回和非流式返回有什么区别？
```

它发生在两个层面：

```ABAP
1. dialog_service.py
   async_chat() yield answer / reference / prompt

2. chat_api.py
   session_completion() 接收 async_chat 的 yield，
   格式化为前端需要的数据，
   流式通过 SSE 返回，
   最后更新 ConversationService
```

代码主要在：

api/apps/restful\_apis/chat\_api\.py \(line 1216\)

```Python
async def stream():
    async for ans in async_chat(dia, msg, True, **req):
        ans = _format_answer(ans)
        yield "data:" + json.dumps(...)
    if conv is not None:
        await ConversationService.update_by_id(...)
```

以及非流式：

```Python
answer = None
async for ans in async_chat(dia, msg, False, **req):
    answer = _format_answer(ans)
    if conv is not None:
        await ConversationService.update_by_id(...)
    break
```

---

前面第 6 模块已经生成了：

```ABAP
ans
```

里面可能包含：

```JSON
{
  "answer": "根据员工手册，员工迟到超过 30 分钟需提交说明...",
  "reference": {
    "chunks": [...],
    "doc_aggs": [...]
  },
  "prompt": "...",
  "created_at": 1781000000,
  "final": true
}
```

现在要做两件事：

```ABAP
1. 返回给前端
2. 保存到 conversation
```

---

### 第一步：chat\_api\.py 接收 async\_chat\(\) 的输出。

代码在 api/apps/restful\_apis/chat\_api\.py \(line 1216\)：

```Python
async def stream():
    async for ans in async_chat(dia, msg, True, **req):
        ans = _format_answer(ans)
        yield "data:" + json.dumps({
            "code": 0,
            "message": "",
            "data": ans
        }, ensure_ascii=False) + "\n\n"
```

这里的 async\_chat\(\) 是第 6 模块里的生成器。

它会不断 yield：

```ABAP
中间回答片段
最终完整回答
```

每次 yield 出来，API 层都会包装成 SSE 格式返回给前端。

---

SSE 是什么？

SSE 全称是 Server\-Sent Events，可以理解成：

> 后端不断往前端推送文本片段。
> 
> 

所以前端看到的“打字机效果”，就是这里来的。

最后这个：

```Python
yield "data:" + json.dumps({"code": 0, "message": "", "data": True})
```

表示流式输出结束。

---

### 第二步：格式化答案 \_format\_answer\(\)。

代码在 api/apps/restful\_apis/chat\_api\.py \(line 1209\)：

```ABAP
def _format_answer(ans):
    formatted = structure_answer(conv, ans, message_id, session_id)
    if chat_id:
        formatted["chat_id"] = chat_id
    return formatted
```

这里核心是：

```ABAP
structure_answer(...)
```

它会把 ans 整理成前端需要的结构，同时更新 conv\.message 和 conv\.reference。

---

structure\_answer\(\) 在 api/db/services/conversation\_service\.py \(line 73\)

第一步取出引用：

```ABAP
reference = ans["reference"]
```

然后判断这是不是最终回答：

```ABAP
is_final = ans.get("final", True)
```

流式中间片段一般是：

```ABAP
final = False
```

最后完整回答是：

```ABAP
final = True
```

---

### 第三步：整理引用 chunks。

```ABAP
chunk_list = chunks_format(reference)
reference["chunks"] = chunk_list
```

chunks\_format\(\) 会把检索结果里的 chunk 转成统一前端格式。

比如原始 chunk 可能是：

```JSON
{
  "chunk_id": "chunk_001",
  "content_with_weight": "员工迟到超过 30 分钟需提交说明...",
  "doc_id": "doc_001",
  "docnm_kwd": "员工手册.pdf",
  "kb_id": "kb_001",
  "positions": [...]
}
```

会整理成：

```JSON
{
  "id": "chunk_001",
  "content": "员工迟到超过 30 分钟需提交说明...",
  "document_id": "doc_001",
  "document_name": "员工手册.pdf",
  "dataset_id": "kb_001",
  "positions": [...]
}
```

这样前端不用关心后端索引库字段名。

---

### 第四步：给回答补 ID 和 session\_id。

```ABAP
ans["id"] = message_id
ans["session_id"] = session_id
```

所以前端拿到的每条 assistant 消息都有：

```ABAP
message_id
session_id
chat_id
```

方便后面做：

```ABAP
删除消息
反馈 thumbs up/down
查看引用
继续对话
```

---

### 第五步：更新 conversation\.message。

代码在 api/db/services/conversation\_service\.py \(line 97\)：

```SQL
if not conv.message or conv.message[-1].get("role", "") != "assistant":
    conv.message.append({
        "role": "assistant",
        "content": content,
        "created_at": time.time(),
        "id": message_id
    })
```

如果当前 conversation 末尾还没有 assistant 消息，就新增一条。

流式中间片段时，会走：

```ABAP
conv.message[-1]["content"] = (
    conv.message[-1].get("content") or ""
) + content
```

也就是说，流式过程中：

```ABAP
第一次片段: "员工"
第二次片段: "员工迟到"
第三次片段: "员工迟到超过..."
```

它是在内存里的 conv\.message\[\-1\]\["content"\] 逐步追加。

---

### 第六步：最终回答会覆盖 assistant 消息。

代码在 api/db/services/conversation\_service\.py \(line 100\)：

```JSON
if is_final:
    if ans.get("answer"):
        conv.message[-1] = {
            "role": "assistant",
            "content": ans["answer"],
            "created_at": time.time(),
            "id": message_id
        }
```

为什么最终要覆盖？

因为流式增量过程中可能只是临时拼接，最终 decorate\_answer\(\) 后的答案可能已经被：

```ABAP
补了引用
修复了 citation
处理了 think 标签
清理了格式
```

所以最终以完整答案为准。

---

### 第七步：更新 conversation\.reference。

代码在 api/db/services/conversation\_service\.py \(line 110\)：

```ABAP
if conv.reference:
    should_update_reference = (
        is_final
        or bool(reference.get("chunks"))
        or bool(reference.get("doc_aggs"))
    )
    if should_update_reference:
        conv.reference[-1] = reference
```

还记得第二模块里，每轮新问题都会先创建一个引用槽位：

```ABAP
conv.reference.append({"chunks": [], "doc_aggs": []})
```

现在最终回答出来后，就把本轮引用写进去：

```ABAP
conv.reference[-1] = 当前回答的 reference
```

所以 conversation 里是这样对应的：

```ABAP
message[用户问题]
message[助手回答]
reference[-1] 这一轮助手回答引用的 chunks
```

这就是前端能展示“回答引用了哪些文档片段”的原因。

---

### 第八步：流式返回结束后，持久化到 MySQL。

代码在 api/apps/restful\_apis/chat\_api\.py \(line 1223\)：

```ABAP
if conv is not None:
    await thread_pool_exec(
        ConversationService.update_by_id,
        conv.id,
        conv.to_dict()
    )
```

注意，流式过程中每个片段都更新的是内存里的 conv 对象。

等整个 stream 结束以后，才把完整的 conversation 写回 MySQL。

写入内容包括：

```ABAP
conv.message
conv.reference
```

也就是：

```ABAP
用户问题
助手答案
引用 chunks
引用 doc_aggs
```

---

如果不是流式，逻辑更简单。

代码在 api/apps/restful\_apis/chat\_api\.py \(line 1238\)：

```Python
answer = None

async for ans in async_chat(dia, msg, False, **req):
    answer = _format_answer(ans)
    if conv is not None:
        await thread_pool_exec(
            ConversationService.update_by_id,
            conv.id,
            conv.to_dict()
        )
    break

return get_json_result(data=answer)
```

非流式就是：

```ABAP
等完整答案生成
structure_answer 更新 conv
直接保存 conv
一次性返回 JSON
```

### 第九步：异常返回。

流式里如果出错：

```Python
except Exception as ex:
    logging.exception(ex)
    yield "data:" + json.dumps({
        "code": 500,
        "message": str(ex),
        "data": {
            "answer": "**ERROR**: " + str(ex),
            "reference": []
        }
    })
```

所以前端也会收到错误消息，不是连接直接断掉无响应。

---

第 7 模块负责把 LLM 生成结果包装成前端协议，同时把 assistant 消息和引用来源写回 conversation，使这次问答成为可继续、可回看、可追溯的一轮会话。
