## 六\. Prompt 构造与 LLM 生成。

上一模块结束时，RAGFlow 已经准备好了：

```ABAP
kwargs["knowledge"]
```

也就是检索出来的知识库上下文。

现在第 6 模块要做的是：

```ABAP
把 knowledge 填进 system prompt
拼上历史对话
控制 token 长度
调用 LLM 生成回答
生成后补引用
```

代码从 api/db/services/dialog\_service\.py \(line 763\) 开始。

### 第二步，取出模型生成配置。


这个 dialog\.llm\_setting 是 Chat 配置里的 LLM 参数，比如：

### 第三步，构造 system prompt。

代码在 api/db/services/dialog\_service\.py \(line 766\)：

```ABAP

        "content": prompt_config["system"].format(**kwargs) + attachments_

```

这里最关键的是：

```ABAP

```

如果原来的 system prompt 是：

```ABAP
你是 HR 助手，请基于以下知识库回答问题：

{knowledge}
```

那么 \.format\(\*\*kwargs\) 会把 \{knowledge\} 替换成刚刚拼好的知识上下文：

```ABAP
你是 HR 助手，请基于以下知识库回答问题：

------
ID: 0
Title: 员工手册.pdf
Content:
员工迟到超过 30 分钟需提交说明。

------
ID: 1
Title: 考勤制度.pdf
Content:
迟到三次以上需由直属主管约谈。
```

这就是 RAG 的核心动作：这就是 RAG 的核心动作：

```ABAP
检索结果 -> 塞进 prompt -> 让 LLM 基于它回答
```

### 第四步，可选加入引用提示词。

代码在 api/db/services/dialog\_service\.py \(line 767\)：


如果开启引用：


### 第五步，拼接历史对话。

代码在 api/db/services/dialog\_service\.py \(line 770\)：

```Python
msg.extend([
    {
        "role": m["role"],
        "content": re.sub(r"##\d+\$\$", "", m["content"])
    }
    for m in messages
    if m["role"] != "system"

```

现在 msg 会变成：

```JSON
[
  {
    "role": "system",
    "content": "你是 HR 助手...这里是知识库..."
  },
  {
    "role": "user",
    "content": "员工迟到怎么处理？"
  }
]
```

如果是多轮对话，可能是：

```JSON
[
  {"role": "system", "content": "...knowledge..."},
  {"role": "user", "content": "员工年假怎么规定？"},
  {"role": "assistant", "content": "根据员工手册..."},
  {"role": "user", "content": "试用期也有吗？"}
]
```

注意，这里会清掉回答里已有的引用标记：

```Python

```

避免历史回答里的引用格式干扰下一轮。

### 第六步，控制 token 长度。

代码在 api/db/services/dialog\_service\.py \(line 771\)：

```ABAP
used_token_count, msg = message_fit_in(msg, int(max_tokens * 0.95))
```

message\_fit\_in\(\) 在 rag/prompts/generator\.py \(line 68\)。

它的作用是：

如果 prompt 太长，就裁剪，保证不会超过模型上下文窗口。

为什么需要这一步？

因为 system prompt 里已经塞入了知识库 chunks，加上历史对话，很容易超出模型长度限制。

它会优先保留：

```ABAP
system 消息
最后一条用户消息
```

必要时裁剪内容。

所以 RAGFlow 不会无限把所有历史和所有 chunks 都塞给模型。

---

### 第七步，如果有图片附件，转成多模态消息。

代码在 api/db/services/dialog\_service\.py \(line 772\)：

```ABAP
if llm_type == "chat" and image_attachments:
    convert_last_user_msg_to_multimodal(msg, image_attachments, factory)
```

如果用户本轮上传了图片，并且模型支持多模态，它会把最后一个 user message 转成包含图片的格式。

普通文本问答可以先忽略这一支。

### 第八步，调整 max\_tokens。

代码在 api/db/services/dialog\_service\.py \(line 777\)：

```ABAP
if "max_tokens" in gen_conf:
    gen_conf["max_tokens"] = min(
        gen_conf["max_tokens"],
        max_tokens - used_token_count
    )
```

意思是：

```ABAP
模型总上下文长度 = 输入 prompt + 输出 answer
```

如果输入已经用了很多 token，那输出最多能生成的 token 就要减少。

比如模型最大 8192 token：

```ABAP
输入 prompt 用了 6000
那输出最多不能超过 2192
```

这一步是为了防止请求超模型限制。

### 第九步，定义 decorate\_answer\(\)。

代码从 api/db/services/dialog\_service\.py \(line 780\) 开始：

```ABAP
async def decorate_answer(answer):
```

这个函数不是生成答案，而是生成完成后的后处理。

它会做：

```ABAP
1. 处理 </think>
2. 插入或修复引用
3. 整理 reference
4. 删除 vector，避免返回给前端
5. 记录耗时和 token
6. 返回最终 answer / reference / prompt
```

这一部分其实连接第 7 模块，但它在第 6 模块里定义，所以我们先知道它存在。

### 第十步，调用 LLM，流式生成。

代码在 api/db/services/dialog\_service\.py \(line 883\)：

```ABAP
if stream:
    if llm_type == "chat":
        stream_iter = chat_mdl.async_chat_streamly_delta(
            prompt + prompt4citation,
            msg[1:],
            gen_conf
        )
```

这里非常关键。

传给 LLM 的参数分成两部分：

```ABAP
system = prompt + prompt4citation
history = msg[1:]
```

也就是：

```ABAP
system:
  包含系统指令 + knowledge + 引用规则

history:
  用户/助手历史消息，不含 system
```

chat\_mdl 是 LLMBundle，它屏蔽了底层模型厂商差异。

它最终会去调用不同厂商模型，比如：

```ABAP
OpenAI-compatible
Qwen
DeepSeek
Ollama
Claude
Gemini
本地模型
```

上层统一调用：

```ABAP
chat_mdl.async_chat_streamly_delta(...)
```

---

LLMBundle 的封装在 api/db/services/llm\_service\.py \(line 489\)：

```ABAP
async def async_chat_streamly_delta(self, system, history, gen_conf, **kwargs):
```

它会判断底层模型有没有：

```ABAP
async_chat_streamly
```

然后逐步 yield 文本片段：

```ABAP
async for txt in chat_partial(...):
    ans += txt
    yield txt
```

所以这就是流式输出的来源。

LLM 每生成一点：

```ABAP
员工迟到...
```

后端就 yield 一点给前端。

### 第十二步，生成结束后做最终装饰。

代码在 api/db/services/dialog\_service\.py \(line 896\)：

```SQL
full_answer = last_state.full_text if last_state else ""

if full_answer:
    final = await decorate_answer(...)
    final["final"] = True
    final["audio_binary"] = None
    yield final
```

也就是说，流式输出会有两类消息：

```ABAP
中间消息:
  final = False
  只有 answer 增量

最终消息:
  final = True
  包含完整 answer
  包含 reference
  包含 prompt
```

这就是为什么前端可以边显示回答，最后再显示引用来源。

如果不是流式，就走 api/db/services/dialog\_service\.py \(line 903\)：

```ABAP
answer = await chat_mdl.async_chat(
    prompt + prompt4citation,
    msg[1:],
    gen_conf
)

res = await decorate_answer(answer)
yield res
```

非流式就是：

```ABAP
一次性等 LLM 生成完整 answer
然后 decorate_answer
然后返回
```
