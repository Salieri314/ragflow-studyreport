## 一\. Chat 请求入口

### 最典型入口在：

api/apps/restful\_apis/chat\_api\.py \(line 1126\)

```Python
@manager.route("/chat/completions", methods=["POST"])
async def session_completion(chat_id_in_arg=""):
```

也就是说，当前端用户在 Chat 界面里输入问题并发送，最终会打到类似这个接口：

```ABAP
POST /chat/completions
```

这个接口就是普通 Chat RAG 问答的主入口。

它大概接收这样的请求：

```JSON
{
  "chat_id": "chat_001",
  "question": "员工迟到怎么处理？",
  "session_id": "session_001",
  "stream": true
}
```

不同调用方式字段可能略有差异，但核心一定有：

```ABAP
用户问题
chat/dialog ID
session/conversation ID
是否流式输出
```

---

这个入口的职责不是马上检索，而是先做几件“接待工作”：

```ABAP
1. 读取请求 JSON
2. 确认 chat_id / session_id
3. 确认用户有权限访问这个 chat
4. 找到或创建 conversation session
5. 把用户问题整理成内部 message 格式
6. 准备后续生成器
```

所以第 1 模块可以理解为：

把前端的一次提问，转成 RAGFlow 后端能够继续处理的一次会话请求。



它还要区分两种情况。

### 情况一：已有会话

比如用户已经和某个 Chat 机器人聊过几轮：

```ABAP
session_id = session_001
```

后端就会把这轮问题追加到已有 conversation 里。

### 情况二：新会话

如果没有 session\_id，后端会创建一个新的 conversation：

```ABAP
新建 session
保存初始 message
绑定 chat_id / user_id
```

这就是为什么 RAGFlow 可以在左侧看到历史对话列表。
