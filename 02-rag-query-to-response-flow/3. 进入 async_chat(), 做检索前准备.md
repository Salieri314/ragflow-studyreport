## 三\. 进入 async\_chat\(\)，做检索前准备

上一部分结束时，chat\_api\.py 已经拿到了两个核心对象：

```ABAP
dia   # Chat / Dialog 配置
msg   # 本轮对话消息
```

然后它会调用：

```ABAP
async for ans in async_chat(dia, msg, True, **req):
```

真正进入 RAG 问答核心。入口在 api/db/services/dialog\_service\.py \(line 546\)：

```ABAP
async def async_chat(dialog, messages, stream=True, **kwargs):
```

这里的 dialog 就是上一部分的 dia，里面有：

```ABAP
kb_ids
llm_id
rerank_id
prompt_config
top_n
top_k
similarity_threshold
vector_similarity_weight
meta_data_filter
```

messages 是当前会话历史，最后一条必须是用户问题。

---

### 第一步，确认最后一条消息来自用户。

```ABAP
assert messages[-1]["role"] == "user"
```

这是一种保护。

因为 RAG 问答必须以用户问题作为触发点。如果最后一条不是 user，后面就不知道该拿什么去检索。

---

### 第二步，判断是不是“无知识库直聊”。

代码在 api/db/services/dialog\_service\.py \(line 549\)：

```ABAP
use_web_search = _should_use_web_search(dialog.prompt_config, kwargs.get("internet"))

if not dialog.kb_ids and not use_web_search:
    async for ans in async_chat_solo(dialog, messages, stream):
        yield ans
    return
```

意思是：

```ABAP
如果这个 Chat 没绑定知识库
也没开启联网搜索
那就不走 RAG
直接让 LLM 聊天
```

也就是说，RAGFlow 的 Chat 有两种模式：

```ABAP
普通 LLM Chat:
  用户问题 -> LLM

RAG Chat:
  用户问题 -> 检索知识库 -> LLM
```

我们分析的是第二种，所以必须满足：

```ABAP
dialog.kb_ids 不为空
```

或者启用了 web search。

---

### 第三步，加载 LLM 模型信息。

代码在 api/db/services/dialog\_service\.py \(line 557\)：

```ABAP
llm_type = TenantLLMService.llm_id2llm_type(dialog.llm_id)
```

RAGFlow 会先判断这个 llm\_id 对应哪类模型。

可能是：

```ABAP
chat
image2text
```

然后读取模型配置：

```ABAP
llm_model_config = TenantLLMService.get_model_config(
    dialog.tenant_id,
    LLMType.CHAT,
    dialog.llm_id
)
```

并拿到：

```ABAP
factory = llm_model_config.get("llm_factory", "")
max_tokens = llm_model_config.get("max_tokens", 8192)
```

这里的 max\_tokens 后面会决定 prompt 最多能塞多少内容。

也就是说，检索出来的 chunks 不是无限塞进 prompt，而是要受模型上下文长度限制。

---

### 第四步，加载四类模型对象。

代码在 api/db/services/dialog\_service\.py \(line 584\)：

```ABAP
kbs, embd_mdl, rerank_mdl, chat_mdl, tts_mdl = get_models(dialog)
```

get\_models\(\) 在 api/db/services/dialog\_service\.py \(line 343\)。

它会根据 dialog 配置加载：

```ABAP
kbs:
  知识库对象列表

embd_mdl:
  embedding 模型，用于把用户问题转成向量

rerank_mdl:
  rerank 模型，可选，用于重排召回 chunks

chat_mdl:
  chat LLM，用于最终生成答案

tts_mdl:
  语音合成模型，可选
```

这里有个特别重要的检查：

```ABAP
embedding_list = list(set([kb.embd_id for kb in kbs]))
if len(embedding_list) > 1:
    raise Exception("Knowledge bases use different embedding models.")
```

意思是：一个 Chat 如果绑定多个知识库，这些知识库必须使用同一个 embedding 模型。

为什么？

因为用户问题会被 encode 成一个向量，然后去多个知识库里检索。

如果知识库 A 用 768 维 embedding，知识库 B 用 1024 维 embedding，那同一个查询向量没法同时正确匹配它们。

所以这里要求：

```ABAP
多个知识库 embedding 模型一致
```

---

### 第五步，取最近几个用户问题。

代码在 api/db/services/dialog\_service\.py \(line 591\)：

```ABAP
questions = [m["content"] for m in messages if m["role"] == "user"][-3:]
```

它不是只看最后一句，而是最多取最近 3 个用户问题。

比如对话是：

```ABAP
用户：公司有年假吗？
助手：有。
用户：试用期有吗？
助手：...
用户：能休几天？
```

如果只拿最后一句：

```ABAP
能休几天？
```

其实很难检索，因为不知道问的是“什么能休几天”。

所以 RAGFlow 会保留最近几轮用户问题，为后面的“多轮问题改写”做准备。

---

### 第六步，处理附件。

代码在 api/db/services/dialog\_service\.py \(line 592\) 到 605。

它支持两类附件：

```ABAP
doc_ids:
  指定只在某些文档里检索

files:
  用户本轮上传的临时文件或图片
```

如果请求里有：

```ABAP
doc_ids
```

就会得到：

```ABAP
attachments = [...]
```

后面检索时会传给：

```ABAP
doc_ids=attachments
```

这表示：

> 本次检索限制在指定文档范围内。
> 
> 

如果消息里有 files，它会把文件拆成：

```ABAP
text_attachments
image_attachments
image_files
```

文本附件会拼到后面 prompt 里：

```ABAP
attachments_ = "\n\n".join(text_attachments)
```

图片附件则可能走多模态模型处理。

---

### 第七步，读取 prompt 配置。

```ABAP
prompt_config = dialog.prompt_config
```

这个配置很重要，里面可能有：

```YAML
system:
  系统提示词模板

parameters:
  需要填入 prompt 的变量

  没检索到内容时的回答

  是否多轮问题改写

  是否跨语言改写

  是否给问题补关键词

use_kg:
  是否使用知识图谱检索

toc_enhance:
  是否用目录增强
```

也就是说，prompt\_config 既控制生成，也控制部分检索前处理

---

### 第八步，尝试 SQL 检索分支。

代码在 api/db/services/dialog\_service\.py \(line 609\)：

```ABAP
field_map = KnowledgebaseService.get_field_map(dialog.kb_ids)
```

如果知识库有字段映射，RAGFlow 会先尝试：

```ABAP
ans = await use_sql(...)
```

这主要是为了结构化/表格类知识库。

例如用户问：

```ABAP
2024 年销售额总和是多少？
```

这类问题如果能转成 SQL 聚合查询，可能比向量检索更准确。

如果 SQL 得到了有效答案，就直接：

```ABAP
yield ans
return
```

如果 SQL 没有结果，才回退到普通向量/全文检索

```ABAP
logging.debug("SQL failed or returned no results, falling back to vector search")
```

所以这里有一个隐藏分支：

```ABAP
如果知识库适合 SQL 查询:
  先试 SQL
失败后:
  回退普通 RAG 检索
```

---

### 第九步，确认 prompt 里是否需要 \{knowledge\}。

代码在 api/db/services/dialog\_service\.py \(line 630\)：

```SQL
param_keys = [p["key"] for p in prompt_config.get("parameters", [])]

if dialog.kb_ids and "knowledge" not in param_keys and "{knowledge}" in prompt_config.get("system", ""):
    prompt_config.setdefault("parameters", []).append({"key": "knowledge", "optional": False})
```

这一步是为了保证后面能把检索到的知识塞进 prompt。

典型 system prompt 是：

```ABAP
你是智能助手，请基于以下知识库回答：
{knowledge}
```

其中 \{knowledge\} 就是占位符。

如果 prompt 里有 \{knowledge\}，但参数列表里漏了 "knowledge"，RAGFlow 会自动补上。

---

### 第十步，处理其他必填 prompt 参数。

代码在 api/db/services/dialog\_service\.py \(line 637\)：

```Python
for p in prompt_config.get("parameters", []):
    if p["key"] == "knowledge":
        continue
    if p["key"] not in kwargs and not p["optional"]:
        raise KeyError("Miss parameter: " + p["key"])
    if p["key"] not in kwargs:
        prompt_config["system"] = prompt_config["system"].replace("{%s}" % p["key"], " ")
```

意思是：

```ABAP
如果 prompt 里有自定义变量，比如 {company_name}
请求必须传入它
如果变量是 optional 且没传，就替换为空
```

例如 prompt 是：

```ABAP
你是 {company_name} 的 HR 助手。
```

那么请求必须带：

```ABAP
{
  "company_name": "某某公司"
}
```

否则会报缺参数。

---

### 第十一步，多轮问题改写。

代码在 api/db/services/dialog\_service\.py \(line 645\)：

```ABAP
if len(questions) > 1 and prompt_config.get("refine_multiturn"):
    questions = [await full_question(dialog.tenant_id, dialog.llm_id, messages)]
else:
    questions = questions[-1:]
```

这个非常重要。

如果用户连续问：

```ABAP
用户：员工手册里年假怎么规定？
助手：...
用户：试用期也有吗？
```

最后一句：

```ABAP
试用期也有吗？
```

单独拿去检索很弱。

所以 RAGFlow 会调用 LLM 把它改写成完整问题：

```ABAP
员工手册中，试用期员工是否享有年假？
```

这就是 full\_question\(\) 的作用。

所以检索前的问题可能不是用户原句，而是被改写后的完整问题。

---

### 第十二步，跨语言改写。

代码在 api/db/services/dialog\_service\.py \(line 650\)：

### 第十四步，关键词增强。

代码在 api/db/services/dialog\_service\.py \(line 664\)：

```ABAP
if prompt_config.get("keyword", False):
    questions[-1] = questions[-1] + "," + await keyword_extraction(chat_mdl, questions[-1])
```

如果开启了 keyword，RAGFlow 会让 LLM 从问题里提取关键词，然后拼回问题。

比如：

```ABAP

关键词：考勤, 迟到, 处罚
```

最终检索 query 可能变成：

```ABAP
员工迟到怎么处理？,考勤,迟到,处罚
```

这对全文检索会有帮助。
