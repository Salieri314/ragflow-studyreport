## 四\. settings\.retriever\.retrieval\(\) 混合检索

上一模块最后，async\_chat\(\) 已经准备好了：

```ABAP
questions
embd_mdl
rerank_mdl
dialog.kb_ids
dialog.top_n
dialog.top_k
dialog.similarity_threshold
dialog.vector_similarity_weight
attachments / doc_ids
```

然后真正进入检索。

入口在 api/db/services/dialog\_service\.py \(line 715\)：

```ABAP
kbinfos = await retriever.retrieval(
    " ".join(questions),
    embd_mdl,
    tenant_ids,
    dialog.kb_ids,
    1,
    dialog.top_n,

    dialog.vector_similarity_weight,

    top=dialog.top_k,
    aggs=True,

    rank_feature=label_question(" ".join(questions), kbs),

```

这行是 Chat RAG 召回的核心。

---

### 先解释这些参数。

```ABAP
" ".join(questions)
```

这是最终检索问题。

它可能已经经过了上一模块的多轮改写、跨语言改写、关键词增强。

比如用户原来问：

```ABAP
试用期也有吗？
```

经过改写后可能变成：

```ABAP
员工手册中，试用期员工是否享有年假？
```

真正用于检索的是后者。

---

```ABAP
embd_mdl
```

这是 embedding 模型。

它负责把用户问题变成向量。

第一条链路里，chunk 入库时已经被 embedding 成：

```ABAP
q_1024_vec
```

现在用户问题也要用同一个 embedding 模型转成向量，才能做向量相似度检索。

---

```ABAP
tenant_ids
dialog.kb_ids
```

这是检索范围。

```ABAP
tenant_ids:
  查哪些租户的索引

kb_ids:
  查哪些知识库
```

RAGFlow 的索引库不是随便全局搜，它会被限制在当前 Chat 绑定的知识库里。

---

```ABAP
dialog.top_n
dialog.top_k
```

这两个容易混。

top\_k 是初筛候选数量，默认可能很大，比如：

```ABAP
1024
```

意思是从索引库里最多先拿多少候选。

top\_n 是最终返回给 LLM 的 chunks 数，比如：

```ABAP
6
```

所以流程是：

```ABAP
先从索引库召回 top_k 候选
再排序 / rerank / 过滤
最后取 top_n 放进 prompt
```

---

```ABAP
dialog.similarity_threshold
```

这是相似度阈值。

如果 chunk 分数太低，就不返回。

比如阈值是：

```ABAP
0.2
```

那低于 0\.2 的 chunk 会被过滤掉。

---

```ABAP
dialog.vector_similarity_weight
```

这是向量相似度权重。

RAGFlow 做的是混合检索，不是纯向量，也不是纯关键词。

它会计算：

```ABAP
最终分数 = 全文关键词分数 * 文本权重 + 向量分数 * 向量权重
```

其中：

```ABAP
向量权重 = vector_similarity_weight
全文权重 = 1 - vector_similarity_weight
```

比如：

```ABAP
vector_similarity_weight = 0.3
```

就表示：

```ABAP
向量分数占 30%
关键词/全文分数占 70%
```

这和很多只做纯向量的 RAG 系统不一样。

---

```ABAP
doc_ids=attachments
```

如果用户指定了某些文档，就只在这些文档里搜。

例如用户问：

```ABAP
只看这份员工手册，迟到怎么处理？
```

那检索范围可以被限制在某几个 doc\_id 里。

---

```ABAP
rerank_mdl=rerank_mdl
```

如果 Chat 配置了 rerank 模型，就把初筛结果再交给 rerank 模型重排。

比如：

```ABAP
初筛召回 64 个 chunk
rerank 模型判断哪几个最贴近问题
最终取 top_n
```

---

### 第五步，生成向量检索表达式。

如果有 embedding 模型，会执行 rag/nlp/search\.py \(line 181\)：

```ABAP

```

get\_vector\(\) 在 rag/nlp/search\.py \(line 53\)：


这里做了几件事：

```ABAP

```

比如用户问题向量是 1024 维，那么字段名就是：

```ABAP

```

这正是第一条链路写入 chunk 时生成的向量字段。

所以这里是第一条线和第二条线最核心的衔接点：

```ABAP
第一条线:
  chunk -> q_1024_vec -> 写入索引库

第二条线:
  question -> q_1024_vec -> 去索引库查相似 chunks
```

### ES / Infinity / OceanBase的区别

RAGFlow 会在 ES / Infinity / OceanBase 里选择一个作为 doc store / 索引库使用，它们在 RAGFlow 里承担的是：

```ABAP
chunk 索引库
= 存 chunk 文本
+ 存全文检索字段
+ 存向量字段
+ 支持过滤 / 排序 / 聚合
```

所以它不只是“向量库”，而是更像：

全文检索 \+ 向量检索的混合索引库。



这三个的共同点是：

```ABAP
都可以保存 chunks
都可以保存 embedding 向量
都可以被 RAGFlow 用来做检索
都不是 MySQL
```

第一条链路里：

```ABAP
chunk + q_xxx_vec -> 写入 ES / Infinity / OceanBase
```

第二条链路里：

```ABAP
用户问题 -> 向量 -> 去 ES / Infinity / OceanBase 召回 chunk
```

---

区别大概是：

```ABAP
Elasticsearch:
  传统全文检索强
  生态成熟
  也支持向量检索
  RAGFlow 里常作为默认/常见 doc store

Infinity:
  更偏 AI-native 的检索引擎
  对全文 + 向量混合检索支持更直接
  RAGFlow 代码里对 Infinity 有专门优化

OceanBase:
  更偏数据库系统
  在 RAGFlow 里也被适配为 doc store
  可以承载向量/全文/结构化查询能力
```

---

为什么不能只叫“向量数据库”？

因为 RAGFlow 检索时不只查向量。

它会查这些字段：

```ABAP
content_ltks
content_sm_ltks
title_tks
important_kwd
important_tks
question_tks
q_1024_vec    #这个是向量字段
kb_id
doc_id
available_int
page_num_int
position_int
metadata
```

其中

```ABAP
content_ltks
important_kwd
question_tks
title_tks
```

是全文检索字段。

---

三者在代码里通过统一接口屏蔽差异。

RAGFlow 代码里常看到：

```ABAP
settings.docStoreConn
```

这个就是当前配置下的索引库连接。

如果你配置的是 ES：

```ABAP
settings.docStoreConn -> Elasticsearch connector
```

如果你配置的是 Infinity：

```ABAP
settings.docStoreConn -> Infinity connector
```

如果你配置的是 OceanBase：

```ABAP
settings.docStoreConn -> OceanBase connector
```

上层代码一般不直接关心你用哪个：

```ABAP
settings.docStoreConn.insert(...)
settings.docStoreConn.search(...)
settings.docStoreConn.delete(...)
settings.docStoreConn.create_idx(...)
```

也就是说，上层统一调用：

```ABAP
insert / search / delete / create_idx
```

底层由不同实现去适配 ES、Infinity 或 OceanBase。



**ES / Infinity / OceanBase 在 RAGFlow 里是可替换的索引后端，一般选择一个使用；它们承担 chunk 的全文检索和向量检索，不只是单纯向量存储。**



**对 RAGFlow 的主流程来说，选择 ES / Infinity / OceanBase 不改变整体链路；但对检索能力、性能、部署复杂度、某些细节行为会有影响。**
